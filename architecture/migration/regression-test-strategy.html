§5. test 失敗時の運用

  • RED test scaffold pattern 禁止: 新 spec の Inv U で構造的に不能化、新規 test は GREEN で merge
  • RED test 追加が必要な場合: 同 PR 内で実装と pair で merge (RED scaffold 単独 merge 禁止)
  • FAIL 時の対応: ローカル bats <file> で原因特定、必要なら EXP-id 経由 sandbox で reproduce

§6. CI integration

6.1 GitHub Actions workflow 構造案

name: bats regression
on: [push, pull_request]
jobs:
  bats:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup bats
        uses: bats-core/bats-action@v3
        with:
          bats-version: 1.10.0   # >= 1.7.0 (bats_require_minimum_version 対応)
      - name: Verify bash version
        run: bash --version | head -1 | grep -E 'version (5|4\.[3-9])' || exit 1
        # bash 4.3+ 必須 (local -n nameref、EXP-014 verify)
      - name: Run all bats
        run: bats plugins/twl/tests/bats/ test-fixtures/experiments/hooks/
      - name: Run experiments (smoke)
        if: github.event_name == 'pull_request'
        run: bash experiments/run-all.sh

6.2 bats-core v1.7.0+ 規約

新規 bats test は以下を MUST:

  • file 冒頭: bats_require_minimum_version 1.7.0
  • setup / teardown で $SANDBOX = mktemp -d 採用 (既存 common.bash pattern)
  • assertion は組み込み ([ "$status" -eq 0 ]) または bats-assert external (load 必要)
  • run で $status / $output / ${lines[@]} を捕捉
  • 各 EXP-id を bats 化する場合、file 名 = EXP-NNN-<name>.bats

§7. bats-core 公式 verified syntax 引用

WebFetch verified からの引用 bats writing-tests:

bats_require_minimum_version 1.7.0

setup {
  load 'test_helper/bats-support/load'
  load 'test_helper/bats-assert/load'
  SANDBOX=$(mktemp -d)
}

teardown {
  rm -rf "$SANDBOX"
}

@test "EXP-006: mailbox flock atomic write" {
  source plugins/twl/scripts/lib/mailbox.sh
  export TWILL_MAILBOX_ROOT="$SANDBOX/mailbox"

  # 100 並列 write
  for i in $(seq 1 100); do
    mailbox_emit "worker-$i" "pilot" "step-completed" "{\"i\":$i}" &
  done
  wait

  run jq -s 'length' "$SANDBOX/mailbox/pilot/inbox.jsonl"
  assert_success
  assert_output "100"
}
