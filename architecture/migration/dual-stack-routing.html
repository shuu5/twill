目的: administrator (新、案 2 先行実装) と su-observer (旧) が併存する Phase 2 期間中、各 Issue がどちらの flow で処理されるかの判定基準と routing 機構を仕様化する。移行完了 (Phase 3 cutover) 判定基準も明示。

verified context: Agent C 分析 (過去 session) — 「administrator 単独稼働 + pilot/worker 未実装」段階での safe degradation 設計が前提。

§1. routing 判定の SSoT

  • Issue label を SSoT: stack:new / stack:legacy で routing
  • default: 新規 Issue は stack:new (Phase 2 開始時に GitHub Action label-new-stack.yml で自動付与)
  • in-flight Issue は stack:legacy を継続 (手動 migration まで)
  • label が両方 (stack:new + stack:legacy) → CRITICAL、user 介入必要

§3. routing flow

3.1 administrator polling cycle で label を確認

# administrator/SKILL.md Step 1 (polling iteration)
FOR each issue IN board.items:
  IF "stack:legacy" IN issue.labels:
    SKIP  # 旧 su-observer 担当
  IF "stack:new" IN issue.labels:
    PROCESS  # 新 administrator 担当
  IF no stack label:
    DEFAULT = "stack:new"  # 新規 Issue は default 新 stack
    gh issue edit ${issue.num} --add-label "stack:new"
    PROCESS

3.2 su-observer (旧) の routing

su-observer は既存 chain-runner 経由で動作、stack:legacy Issue のみを対象とするように su-observer/SKILL.md の Step 1 polling cycle に label filter 追加 (Phase 2 開始時に minor patch)。

3.3 共存中の制約

  • 両 stack の administrator session (旧 + 新) が並列稼働、tmux window prefix で識別
  • cross-stack mail 禁止: .supervisor/ と .admin/ を相互 read/write 禁止
  • cross-stack spawn 禁止: 旧 su-observer が phaser-* spawn (新 stack 経路) しない、新 administrator が co-autopilot spawn (旧 stack 経路) しない
  • 同 Issue を両 stack が処理すると Project Board status race の risk → label 厳密 enforce + integrity check

§4. migration 経路 (旧 → 新)

4.1 自動 migration (簡単な Issue)

# GitHub Action: migrate-to-new-stack.yml
ON: issue label "migrate-to-new" added by user
STEPS:
  1. Verify issue is in safe transition state (Status NOT IN [Implementing, PR Reviewed])
  2. Remove "stack:legacy", add "stack:new"
  3. Remove old .autopilot/checkpoints/issue-${num}.json (旧 checkpoint clear)
  4. Notify administrator via .mailbox/administrator/inbox.jsonl

4.2 手動 migration (in-flight Issue)

Implementing / PR Reviewed 状態の Issue は手動 migration が必要 (新 stack で state 再構築):

  1. user が gh issue edit で label 切替
  2. 旧 stack の関連 tmux window を kill (su-observer に手動 instruction)
  3. 新 stack の administrator が CronCreate next cycle で検知 → phaser-impl 再開
  4. 移行 cost: 1 Issue あたり 約 10 min (user 手動作業含む)

§5. Phase 2 期間中の制約

  • 新規 Issue は新 stack 必須: label-new-stack.yml Action で stack:new default 付与
  • 旧 stack は freeze: 新規修正禁止、bug fix 以外の改修 PR を Action で block (freeze-legacy-stack.yml)
  • 両 stack の bats test を CI で並列実行: 旧 chain-runner と新 step.sh の整合性を継続 verify
  • budget 配分: 5h budget は both stack で共有、片方が暴走しないよう MAX_PARALLEL_PILOTS を厳格に

§6. Phase 3 cutover 判定基準

  1. 旧 stack の in-flight Issue が全て Merged or 新 stack 移行完了
  2. 新 stack で 10+ Issue が安定完遂 (PR merge 成功率 > 90%)
  3. bats regression 全 PASS (新 + 旧の test suite)
  4. Phase 1 PoC verify points (VP-1〜VP-12) 全件 PASS
  5. user 明示承認 (Phase 3 cutover は user 明示承認必須、autonomous 判断不可)

§7. cutover 後の cleanup

  • 旧 stack file 削除 (deletion-inventory.html に従う): chain-runner.sh (1714 行) / spawn-controller.sh (778 行) / wave-progress-watchdog.sh (281 行) 等
  • stack: label を全 Issue から削除 (script 1 回実行)
  • administrator (旧 su-observer rebrand) のみ稼働、co-* は archive
  • .supervisor/ directory を .archive/supervisor-${date}/ に移動 (rollback 用に保全)
