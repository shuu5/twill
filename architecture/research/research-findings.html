§0. エグゼクティブ・サマリー (5 件、verified)

  1. twl MCP は thin helper として維持・縮小、公式 github/github-mcp-server に大半を委譲。Branch protection 操作と Issue → Project item linkage の自動化のみ twl 側で wrapper 化する価値あり。
  2. 「1 行コマンド = 1 tool」薄ラッパーは Anthropic 公式推奨に反する。MCP は CLI 単独より 35x token 重く、reliability も 100% → 72% に低下する benchmark あり。workflow 統合型 tool (twl_issue_promote 等) を coarse-grained に設計し、tool 総数 20 前後を目標。
  3. mcp-watchdog (Bug #1687) は廃止が最適。Claude Code は stdio MCP を auto-reconnect しない仕様 (Issue #43177)、サーバー側 watchdog は orphan 自殺で逆効果 (Issue #1478)。代替は ① HTTP transport 移行 ② Plugin Monitors (v2.1.105+) ③ stdin EOF で素直に exit + 必要時 /reload-plugins 手動 の 3 経路。
  4. PreToolUse hook + MCP shadow tier は公式仕様で実現可能。tier 1 = command hook (粗フィルタ + daemon spawn 不可)、tier 2 = mcp_tool hook (接続済み MCP で stateful 判定) を並列実行、deny > ask > allow の最 restrictive 勝ち、bypassPermissions でも貫通。不変条件 W (status gate 機械化) は実装可能。
  5. tmux と gh CLI は MCP 化より bash helper + PreToolUse hook が現実的。tmux send-keys の escape 罠と tmux -C kill-server の破壊性は wrapper で必ず吸収。gh project item-edit --single-select-option-id で status 更新は CLI 単発完結、GraphQL を LLM に書かせる不安定さを排除可能。

KP-2. gh CLI 単発で Project Board status 更新が完結 (GraphQL 不要) [verified]

KP-3. Claude Code stdio MCP は auto-reconnect しない (Bug #1687 根本) [verified]

  • Issue #43177 でソースコード useManageMCPConnections.ts L354-356 で stdio + sdk transport は再接続から明示的に除外
  • HTTP/SSE/WebSocket のみ exponential backoff (5 attempts、1→2→4→8→16s)
  • v2.1.121 で初期接続は 3 回リトライ追加、ただし mid-session disconnect は未対応
  • 修正は 1 行 PR で可能 (suggested、未提出)
source: Issue #43177、Issue #40207

KP-4. サーバー側 watchdog は orphan 自殺問題で逆効果リスク [verified]

  • telegram plugin server.ts L658-664 の orphan watchdog: process.stdin.destroyed 検出で自殺
  • 「外部から respawn しても親 PID 変化で再度 watchdog 自殺の悪循環」と公式 issue で報告
  • MCP プロトコル仕様: 「Servers SHOULD exit promptly when stdin is closed」(stdin EOF = portable graceful shutdown signal)
source: Issue #1478

KP-5. Anthropic 公式は coarse-grained tool を推奨、「1 API endpoint = 1 tool」は anti-pattern [verified]

  • 「Tools can consolidate functionality, handling potentially multiple discrete operations (or API calls) under the hood」
  • 「More tools don't always lead to better outcomes」
  • MindStudio benchmark: MCP は CLI の 35x token を消費、複雑タスクで reliability 100% → 72% に低下
  • 50 tools = 10,000-25,000 tokens (definition のみ、call 前)
source: Anthropic Writing tools for agents

KP-6. PreToolUse hook で status gate 機械化 (不変条件 W) は実装可能 [verified]

  • tier 1 = command hook + if: "Skill(phase-impl*)" 等で粗フィルタ
  • tier 2 = mcp_tool hook で接続済み twl MCP の stateful 判定
  • 同 event の全 hook が parallel 実行、permissionDecision は deny > ask > allow の最 restrictive 勝ち
  • PreToolUse は permission mode より先に発火、bypassPermissions や --dangerously-skip-permissions でも deny は効く
source: Claude Code Hooks reference

KP-7. tmux 操作は LLM の素直な記述で安定しない [verified]

  • send-keys の罠: セミコロン (Issue #4350)、ハイフン先頭 (Issue #4408)、スペース脱落 (Issue #1425)、Escape timing (Discussion #4482)
  • tmux -C (control mode) で kill-server 含む任意 tmux コマンドが実行可能、出力は valid UTF-8 でない可能性あり
  • pane state は capture-pane でなく format 変数 (#{pane_dead}, #{pane_active}, #{pane_in_mode}) が信頼できる
source: tmux Control Mode Wiki

KP-8. MCP の正当化は workflow 統合 / 動的 discovery / 型安全に集約 [verified]

  • 単純な 1 行コマンドラッパーは Bash tool で直接実行で十分 (CLI 100% reliability)
  • MCP 化の価値: ① 複数 CLI を統合した workflow ② Issue 番号 → URL 変換などの前処理 ③ field_id / option_id キャッシュ ④ destructive hint / OAuth などのメタ情報
  • handler 分離パターンは推奨だが「絶対」ではない: FastMCP Client で in-memory test 可能、新規実装で必須ではない
source: Stop Vibe-Testing Your MCP Server、FastMCP Tools docs

運用上の制約 (verified):
  • Prompt injection 脆弱性 (Invariant Labs 2025-05 公表): cross-repository data leakage 攻撃、"no easy solution" 評価
  • 緩和策: --read-only モードまたは信頼 repo 限定運用
  • selective toolset 必須 (--toolsets projects,issues,pull_requests,actions,repos)、copilot discussions 等を無効化して token 削減
  • PAT scope: repo, read:packages, read:org, project (projects toolset 使用時)
  • GITHUB_TOKEN は project scope なし、GitHub Actions で project 操作には PAT 必須

§6. リスク・落とし穴 (6 件、verified)

  1. GitHub 公式 MCP の prompt injection 脆弱性 [verified、★警告]: 2025-05 Invariant Labs 公表、no easy solution。緩和は --read-only + 信頼 repo 限定
  2. addProjectV2ItemById 2026-03 platform regression [uncertain]: success: true でも item が永続化されないバグ報告あり (community discussion #188487)。対策: twl 側で retry + verify
  3. tmux 操作の罠 [verified、wrapper 必須]: send-keys のセミコロン / ハイフン / スペース / Escape / control mode 危険
  4. Claude Code SessionStart hook の制約 [verified]: v2.1.87+ で SessionStart hook が background process spawn すると claude-code が無限 hang (Issue #43123)。対策: nohup ... </dev/null >/dev/null 2>&1 & で完全 fd 切断、または Plugin Monitors
  5. GitHub Actions の SLA 制約 [verified]: cron 最小 5 分、遅延・drop あり、public repo 60 日無活動で無効化
  6. GITHUB_TOKEN の Project Board 操作不可 [verified、★]: project scope なし、PAT 必須
