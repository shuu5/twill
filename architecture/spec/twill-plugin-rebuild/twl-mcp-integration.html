twl MCP 継承戦略 — 新 architecture との接続

draft-v2 (2026-05-12) · 既存 30+ tool + 10 audit sections + TWL_AUDIT 機構の選択的継承
目的: 既存 cli/twl/src/twl/mcp_server/ 配下の 30+ MCP tool、10 audit sections、TWL_AUDIT env + .audit/<run-id>/ 中間ファイル機構、types.yaml SSoT (token 閾値)、tier 階層 (hook + MCP shadow)、deferred-tool 戦略 の継承価値を 1 件ずつ評価し、新 architecture (admin/pilot/worker 3 階層) に組み込む方法を定める。

方針: script 依存を減らす方向ではあるが、理論的骨子に継承価値ありと判断された機能は MCP tool / Python module として継承する。script 形式のままの機能は新 architecture と整合的に MCP 化 or 廃止する。

§1. MCP server 全体構造 (継承する骨子)

1.1 現状 (verified)

既存 twl MCP server は cli/twl/src/twl/mcp_server/ 配下に集約されており、起動経路は:

.mcp.json
  → uv run --directory cli/twl --extra mcp fastmcp run cli/twl/src/twl/mcp_server/server.py
  → server.py: from twl.mcp_server.tools import mcp
  → mcp.run()

tool 登録機構の核心は tools.py 末尾 (line 2311) の deferred-tool パターン:

try:
    from fastmcp import FastMCP
    mcp = FastMCP(...)
except ImportError:
    mcp = None

# 全 handler は *_handler suffix の純 Python 関数として実装
def twl_validate_handler(args):
    ...
    return {"items": [...]}

# MCP wrapper は薄いラッパー
if mcp is not None:
    @mcp.tool()
    def twl_validate(args):
        return json.dumps(twl_validate_handler(args))

この設計により:

  • fastmcp なし環境でも twl_validate_handler() を直接 import して in-process テスト可能
  • handler 関数は副作用最小、入出力は JSON 互換 dict
  • _to_hook_output() (tools.py L1409) は Claude Code 2.1.x の HookOutput Zod schema 変換層で、legacy decision="allow/deny" を "approve/block" に正規化

1.3 継承すべき設計原則 (MUST)

MUST:
  • MCP server は 単一 tools.py + handler 関数 (*_handler suffix) 分離パターン で維持する (in-process テスト可能性の保証)
  • _to_hook_output の HookOutput Zod schema 変換層を維持する (Claude Code 2.1.x+ 互換性)
  • handler 関数は純 Python (副作用最小、入出力 JSON 互換) を MUST

既存 cli/twl/src/twl/validation/audit.py の audit_collect() は 10 section を順に実行し items リストに集約する。各 item は {severity, component, message, section, value, threshold} 形式。本 session の verified 把握:

2.1 10 section の継承判断

#section 名称検証内容新 spec での扱い
1controller_size controller type の本文行数 > 120 (warning) / > 200 (critical) 継承 (新 architecture でも component 品質チェック有効、phase-* / worker-* / tool-* に拡張)
2inline_implementation bash コードブロック行数 / 非空行数比率 > 50% (warning) / > 30% (info) 継承 (bash 削減方針と整合、新 spec で bash inline が増えていないか継続監視)

9chain_integrity deps.yaml workflow calls × chain-runner.sh dispatch table × SKILL.md 実行指示の三者整合性 置換 (新 SSoT 構造に合わせて再実装。ssot-design.html の案 3 step.sh 単一 SSoT に対応する整合性チェックに変換)
10cross_layer_consistency architecture/vision.md の Constraints/Non-Goals が monorepo → cli_twl → plugin_twl 層間で矛盾しないか (ADR-0008) 継承 (新 architecture でも層分離 = cli_twl (Python MCP) + plugin_twl (skill/script) の整合性チェックは有効)

2.2 chain_integrity の置換 (重要)

section 9 chain_integrity の現実装 (audit.py L198-282) は:

  1. chain-runner.sh から step_<func> パターンを _parse_dispatch_table() で抽出 (L105-118)
  2. deps.yaml の workflow skill の calls: リストをイテレート
  3. step: 番号なし + LLM/trigger/SKILL.md 言及なし → warning (orphan_call)
  4. step: あり + chain-runner.sh に step_<func> なし + LLM 宣言なし → critical (dispatch_gap)

これは chain-runner.sh の case 文を読む設計だが、新 architecture では chain-runner.sh が廃止される (ssot-design.html 案 3)。よって section 9 は:

  • 新実装: plugins/twl/scripts/steps/*.sh の filename と deps.yaml.chains の steps list を突合 (1 step = 1 *.sh file = 1 deps.yaml entry の 3-way 整合)
  • 検出: steps/foo.sh 不在 → critical (orphan reference)、steps/foo.sh あり + deps.yaml 不在 → warning (orphan file)

2.3 継承する MUST

MUST:
  • 10 audit sections の枠組みを維持、section 9 のみ新 SSoT 構造に合わせて再実装
  • audit_collect() + audit_report() の 2 段階設計 (収集 + 表示) を継承
  • severity 4 段階 (ok / info / warning / critical) を継承

§3. TWL_AUDIT env + .audit/<run-id>/ 中間ファイル機構

3.1 現状 (verified)

cli/twl/src/twl/autopilot/audit.py が管理する audit mode 機構:

起動条件TWL_AUDIT=1 env OR .audit/.active file 存在
ディレクトリ構造
  • .audit/<run-id>/specialists/ — specialist 出力
  • .audit/<run-id>/checkpoints/ — checkpoint 推移
  • .audit/<run-id>/state-log.jsonl — state 遷移 log
  • .audit/<run-id>/index.json — メタデータ
APIaudit_snapshot(source_dir, label) — isomorphic (audit active 時のみコピー、inactive は no-op)
伝搬launcher が worker spawn 時に TWL_AUDIT=1 -e TWL_AUDIT_DIR=<path> を環境変数として伝搬

3.2 新 architecture での継承

3 階層 (administrator / pilot / worker) での audit propagation モデルとして直接適用可能。継承後の経路:

administrator
  ├── `twl audit on` で `.audit/.active` 作成 + run-id 生成
  ├── pilot spawn 時に TWL_AUDIT=1 TWL_AUDIT_DIR=.audit/<run-id> を env 伝搬
  └── tool-architect / tool-self-improve 等 (独立 tool) も同 env 伝搬

pilot
  ├── env 継承 (TWL_AUDIT, TWL_AUDIT_DIR)
  ├── worker spawn 時に env 伝搬
  └── pilot 自身の review 結果を `.audit/<run-id>/reviews/<issue>.json` に snapshot

worker
  ├── env 継承
  ├── step.sh 内で audit_snapshot(checkpoint_dir, "step-<name>") 呼び出し
  └── 完了時に `.audit/<run-id>/checkpoints/<issue>/` に推移を記録

3.3 継承する MUST + SHOULD

MUST:
  • TWL_AUDIT=1 env + TWL_AUDIT_DIR による audit propagation 機構を新 architecture でも維持
  • audit_snapshot() isomorphic API を維持 (audit active 時のみ動作、inactive で no-op)
  • 新 architecture の per-Worker mailbox (.mailbox/<session>/inbox.jsonl) も audit snapshot 対象に含める
SHOULD:
  • audit mode で administrator polling cycle の全 polling event を log (.audit/<run-id>/admin-cycle.jsonl) に保存
  • audit run の index.json に「どの phase をどの worker が処理したか」の summary を自動生成

§5. tier 階層 (hook + MCP shadow)

5.1 現状 (verified)

現状の検証 / gate 機構は 3 tier 階層:

tier実装responsibility
tier 1bash hook (PreToolUse)fast path (50ms 以内)、結果が確定的、Claude Code 仕様制約への対応
tier 2MCP shadow modetier 1 と同等 logic を MCP tool として登録、in-process テスト可能、Python 型安全
tier 3script (chain-runner.sh 等)BATS テストのみ、MCP 化されていない

例: twl_validate_commit の自己コメント (tools.py L1562) に「files は常に空リストで実質 no-op、bash hook (pre-bash-commit-validate.sh) が本体」と明記。これは MCP hook が staged files を取得できない Claude Code 仕様制約による設計妥協 (Issue #1335 self-deferred)。

5.2 新 architecture での継承

tier 3 (script) は新 architecture で削減方針だが、tier 1 + tier 2 の 2 layer 階層は維持:

新 tier実装新 architecture での responsibility
tier 1PreToolUse hookInv W に基づき phase-* invocation の前提 status を verify。phase-gate.sh (~30 lines) thin helper。fast path (50ms 以内)
tier 2MCP shadow modetier 1 と同等 logic を MCP tool (twl_validate_phase_gate 等) として登録、in-process テスト可能
tier 3step.sh 内の post-verifyInv U に基づき step lifecycle の post-verify phase で機械検証 (test 数 / src diff / RED→GREEN 等)

5.3 継承する MUST

MUST:
  • tier 1 (bash hook) + tier 2 (MCP shadow) の 階層防御を継承
  • Claude Code 仕様制約 (hook が staged files 取得不可等) を前提に設計
  • tier 3 は step.sh 内の post-verify として再設計、chain-runner.sh は廃止

§8. 統合 MUST / SHOULD 一覧

MUST (新 spec で必須)

  1. MCP server を 単一 tools.py + *_handler 関数分離パターンで維持 (in-process テスト可能性)
  2. validation は tier 1 (bash hook) + tier 2 (MCP shadow) の階層防御を継承
  3. TWL_AUDIT=1 env + TWL_AUDIT_DIR による audit propagation 機構を新 architecture でも維持
  4. types.yaml を token 閾値 SSoTとして維持、fallback hardcode を持つ、新型 (administrator / phase-* / worker-* / tool-* / step / subagent) を追加
  5. 10 audit sections の枠組みを維持、section 9 のみ新 SSoT 構造に合わせて再実装
  6. _to_hook_output の HookOutput Zod schema 変換層を維持 (Claude Code 2.1.x+ 互換)

SHOULD (推奨)

  1. chain SSoT を 2-way (step.sh + deps.yaml.chains) に簡素化、chain-steps.sh の bash mirror 廃止 (ssot-design.html)
  2. 不変条件 B (worktree lifecycle Pilot 専任) の MCP gate を新 3 階層 role 機構に統合
  3. audit run の index.json に「どの phase をどの worker が処理したか」の summary を自動生成
  4. token 閾値の types.yaml 値を新コンポーネント型に合わせて再設定

MAY (任意)

  1. twl_audit_session (R1-R4 rules) を新 session 構造に合わせて rule 更新
  2. specialist completeness check (twl_check_completeness / /tmp/.specialist-manifest-*.txt) の manifest ファイル方式を維持するか subagent output 直接読み取りに置換するか選択
  3. MCP tool 一覧を docs に自動生成 (現状は手動 list)
