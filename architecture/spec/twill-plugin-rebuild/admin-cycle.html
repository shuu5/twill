目的

administrator main session の polling cycle (CronCreate 25 min durable で fire) の動作詳細を仕様化する。read 3 (mailbox / GitHub status / tmux windows) → judge → write 3 (spawn / escalate / cleanup) のサイクルと、複数 Issue 並列管理時の conflict 解決を明示する。

想定 outline (次 session で実装)

  1. CronCreate 設定 (25 min durable=true、admin session 起動時に set)
  2. polling cycle 6 step (図 2b、draft-v1 §3b から発展):
    • 1. inbox.jsonl 読込 (未読 mail を line 番号で track)
    • 2. GitHub Project Board status poll (全 Issue Status + 前回 cycle との diff)
    • 3. tmux windows 列挙 (active pilot / worker session 把握)
    • 4. spawn 判断 (Status 遷移 + active 不在 → 次 phase pilot spawn)
    • 5. failure / escalate 処理 (pilot failure mail / human gate)
    • 6. cleanup (completed pilot window close + mailbox archive)
  3. admin-cycle.sh helper 実装 (~30 lines、CronCreate handler)
  4. conflict 解決と不変条件 I-1〜I-7:
    • I-1: 同 Issue に 1 個のみ pilot spawn (idempotent guard)
    • I-2: parallelism 上限 (default 3) 超過時は defer
    • I-3: mail check が next spawn の prerequisite
    • I-4: worker のことを直接知らない (pilot mail のみ)
    • I-5: status 更新は pilot 任せ (admin は読むだけ)
    • I-6: escalate 時は user 確認
    • I-7: tmux send-keys 使用禁止 (mailbox 経由のみ)
  5. pseudo-bash 実装例 (draft-v1 §3b の cycle 関数)
  6. conflict scenario (idempotent / parallelism / stale pilot / race / mail loss / cron skip)
