§2. 共通規約 (4 tool 共通)

2.1 起動経路 (verified、ADR-040 ベース)

toolspawnable_by禁止経路

2.2 mailbox 規約

  • tool-* は status 遷移と無関係、phase-* / worker-* の per-Worker mailbox (.mailbox/<session>/inbox.jsonl) は使用しない
  • tool-self-improve のみ admin への完了 mail 必要 (旧 su-observer 経路の継承、admin-cycle.html §3.1 で受信)
  • tool-architect / tool-project / tool-utility は結果を user に直接提示 (admin mailbox 通知不要)

2.3 TWL_AUDIT 継承

  • 4 tool いずれも user / admin session から env 継承で TWL_AUDIT=1 + TWL_AUDIT_DIR を受け取り、.audit/<run-id>/tools/<tool-name>/ に出力
  • tool 内部の主要 step (PR 作成 / spec 生成 / utility 実行) は audit_snapshot() で snapshot
  • 具体的な出力 schema は twl-mcp-integration.html §3 の TWL_AUDIT 機構を踏襲

2.4 step.sh framework 利用可否

  • tool-* は step.sh framework を利用しない (status 遷移と無関係、step-verification の post-verify 機械検証 (Inv U) は phase-* / worker-* 専用)
  • tool 内部の step 管理は各 tool の SKILL.md で完結 (chain step との混同を避ける)

2.5 PreToolUse gate hook

  • tool-* は gate-hook.html の phase-gate 対象外 (status 遷移と無関係)
  • phase-gate.sh の粗フィルタは if: "Skill(phase-*)" で tool-* を素通り
  • tool-* の起動承認は ADR-040 (user 明示依頼許可) が SSoT

2.6 公式 MCP / twl MCP の利用

  • tool-architect / tool-self-improve は mcp__doobidoo__memory_search を使用 (verified、既存 SKILL.md)
  • tool-* は twl MCP の twl_worktree_create 等を呼ぶ (Phase 5 cleanup 後)
  • GitHub 公式 MCP (research-findings.html §3) は tool-project が大半を使用 (issues / pull_requests / projects toolset)

§4. invocation 経路詳細

4.1 user 直接 invoke

# user が tmux session で直接実行
/twl:tool-architect <feature description>
/twl:tool-project <sub-command> <args>
/twl:tool-utility <command> [args]
/twl:tool-self-improve <scenario|retrospect|test-project-manage>

4.2 admin invoke (tool-architect / tool-self-improve のみ)

administrator (admin-cycle.sh polling) が Board の Idea status Issue を見て tool-architect spawn を判断する場合:

# administrator/SKILL.md Step 1 (polling iteration)
FOR each idea_issue IN board.filter(status="Idea"):
  IF idea_issue.labels contains "feature-dev":
    # ADR-040 gate 通過判定: user 明示 ack を待つ or autonomous (要 user 設定)
    spawn_tmux \
      --window-name "tool-architect-${idea_issue.num}" \
      --skill twl:tool-architect \
      --args "${idea_issue.body}"

4.3 禁止経路 (Inv 関連)

  • phase-* (pilot) が tool-* を Skill() で呼ぶ → MUST NOT (ADR-040 違反)
  • worker-* (worker) が tool-* を Skill() で呼ぶ → MUST NOT
  • cross-stack invoke (旧 su-observer ↔ 新 administrator 経由の tool-* 呼び出し) → dual-stack 期間中 MUST NOT

§7. ADR-040 / ADR-041 との整合性

  • ADR-040 (feature-dev spawn gate): user 明示依頼許可ルール。tool-* の起動承認 SSoT として継承。tool-architect / tool-project / tool-utility / tool-self-improve いずれも user 明示 (slash command) が起動条件
  • ADR-041 (部分 Superseded): spawn-controller.sh 統合は廃止、新 spec では tmux new-window 直接呼び出し (spawn-tmux.sh ~30 行) に簡素化。「user 明示依頼許可」原則は保全

§8. 5 つの曖昧点 (Agent B 提示、本 spec で項目化)

曖昧 1: tool-architect の admin invoke 経路

状況: ADR-040 は「user 明示依頼許可」だが、administrator が Project Board の Idea status Issue を自律的に tool-architect spawn するシナリオが未定義。

選択肢:

  • (A) user 明示のみ (admin spawn 禁止)
  • (B) admin spawn を Idea status + specific label (e.g. auto-feature-dev) で許可
  • (C) admin spawn を user pre-approval (起動時 1 回 ack) で許可

本 spec 暫定方針 (deduced): (A) を default、(B)/(C) は次 session で user 明示確認後に追加

曖昧 2: tool-* の TWL_AUDIT 出力内容

状況: 4 tool が .audit/<run-id>/tools/<name>/ に何を書くか未定義。

暫定方針: tool-architect は spec directory snapshot、tool-project は Board 操作 log、tool-utility は command output、tool-self-improve は scenario 実行 trace を保存 (次 session で詳細化)

曖昧 3: audit section の type 対象拡張

状況: twl MCP の audit section 1 (controller_size > 120 行) が type: tool を対象とするか不明。types.yaml に tool type の token_target を追加するか。

暫定方針 (deduced): types.yaml に tool type を追加 (controller と同じ 1500/2500 token 閾値、120 行 warning は維持)、co-utility (79 行) は当然 PASS

曖昧 4: tool-self-improve の admin mailbox 仕様

状況: 完了 mail の event / detail field 仕様が spawn-protocol.html (outline) に委ねられている。tool-* 専用の mail 仕様 section が必要。

暫定方針: {"from":"tool-self-improve","event":"tool-completed","detail":{"sub_command":"...","result":"..."}} 形式。spawn-protocol.html content 実装時に固める

曖昧 5: tool-utility の commands 拡張性

状況: tool-utility は現在 6 commands のみ、新 architecture で追加 utility を SKILL.md 直接追記 vs catalog 化が未定義。

暫定方針 (deduced): commands < 10 件は SKILL.md 直接記載、超えたら refs/utility-commands.md catalog 化を検討。size-check で 120 行 threshold 接近時に migration 判断
