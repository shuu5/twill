2.2 mailbox 規約

  • tool-sandbox-runner のみ admin への完了 mail 必要 (旧 su-observer 経路の継承、admin-cycle.html §3.1 で受信)
  • tool-architect / tool-project は結果を user に直接提示 (admin mailbox 通知不要)

2.3 TWL_AUDIT 継承

  • 3 tool いずれも user / admin session から env 継承で TWL_AUDIT=1 + TWL_AUDIT_DIR を受け取り、.audit/<run-id>/tools/<tool-name>/ に出力
  • tool 内部の主要 step (PR 作成 / spec 生成 / Board 設定 / sandbox 実行) は audit_snapshot() で snapshot
  • tool 別の出力内容:
    • tool-architect: spec directory snapshot + specialist-spec-review findings JSON
    • tool-project: Board 操作 log + verify result JSON
    • tool-sandbox-runner: sandbox run-id / sandbox log / LLM 分析 result / Idea Issue 起票 list
  • 具体的な出力 schema は twl-mcp-integration.html §3 の TWL_AUDIT 機構を踏襲

2.4 step.sh framework 利用可否

  • tool-* は step.sh framework を利用しない (status 遷移と無関係、step-verification の post-verify 機械検証 (不変条件 U) は phase-* / worker-* 専用)
  • tool 内部の step 管理は各 tool の SKILL.md で完結 (chain step との混同を避ける)

2.5 PreToolUse gate hook

  • tool-* は gate-hook.html の phase-gate 対象外 (status 遷移と無関係)
  • phase-gate.sh の粗フィルタは if: "Skill(phase-*)" で tool-* を素通り
  • tool-* の起動承認は ADR-040 (user 明示依頼許可) が SSoT
  • tool-architect は別系統で spec edit boundary hook を保持 (§3.4 で詳述)

2.6 公式 MCP / twl MCP の利用

  • tool-architect / tool-sandbox-runner は mcp__doobidoo__memory_search を使用 (verified、既存 SKILL.md)
  • tool-* は twl MCP の twl_worktree_create 等を呼ぶ (Phase 5 cleanup 後)

§3. tool-architect 詳細

3.1 役割

  • HTML 形式 architecture spec を 「現仕様」として対話的に維持する
  • feature-dev workflow 経由 (Issue を介さない直接対話)、後の実装指針の大本
  • spec の 唯一 author (本 spec 自体も tool-architect 経由で生成可能、recursive OK)

3.2 verify 強制機構 (2 層併用) EXP-027

層 1: SKILL.md MUST (LLM 規律)
  • Step 0: spec 全 file を grep して 4-state status (inferred / deduced / verified / experiment-verified) を listing、未 verified を todo 化
  • Step N: 新 feature 追加時に EXP 候補生成必須 (experiment-index.html へ追記)
層 2: PostToolUse hook verify-coverage.sh (機械検証)

新規 helper、plugins/twl/scripts/lib/verify-coverage.sh (~50 行)。動作:

  1. grep -c '<span class="vs inferred">' spec/*.html → N inferred (warn)
  2. grep -c '<span class="vs deduced">' spec/*.html → M deduced (warn)
  3. git diff HEAD -- spec/*.html で new section に <a class="exp-link"> 不在を warn
  4. changelog.html を git diff と突合、本日 entry 不在を warn

振る舞い: warn のみ (block せず)、PR review で specialist-spec-review が評価して fix 要求。

2 層必須の根拠 (dig core lesson §5): LLM だけ (層 1) では Step 0 漏れで verify gap、hook だけ (層 2) では context 欠如で false positive。両層併用で漏れと過剰検出の両方を抑制。

3.3 軽量 PR cycle EXP-029

step内容
1feature-dev で spec edit (本 file 自身も含む)
2git commit
3gh pr create --auto-merge=false
4specialist-spec-review spawn (1 agent、新規軽量 review agent、plugins/twl/agents/specialist-spec-review.md ~100 行)
5review findings: 未 verified claim listing / EXP 漏れ check / changelog 漏れ check / dead link check
6tool-architect 自己 fix loop (最大 3 回)
7review PASS or 3 回到達 → user AskUserQuestion approve
8merge (auto-merge は無効、user 明示 approve のみ)

recursive PR rule: tool-architect が自身の spec (tool-architecture.html 等) を更新する PR 作成 OK、ただし review は self skip、user / 他 tool に委譲。

specialist-pr-cycle (8 specialist) との対比: 既存 chain workflow は specialist-pr-cycle で 8 specialist (security/code/llm/...) を回すが、spec PR にはオーバースペック。spec 専用の specialist-spec-review 1 agent に絞ることで軽量化、3 回 fix loop で収束させる。

5.7 doobidoo 重複 check + Idea Issue 起票 EXP-031

flow (semantic search + similarity > 0.75 で link):

for each problem in LLM output:
  query = problem.title + problem.detail (first 200 chars)
  results = mcp__doobidoo__memory_search(query, mode=hybrid)
  if results[0].similarity > 0.75:
    # 重複 → Issue 起票しない、doobidoo hash を log
    log "重複: doobidoo $hash"
    continue
  else:
    # 新規 → Idea Issue 起票
    gh issue create \
      --title "[sandbox/$sandbox_name] $summary" \
      --label "discovered-by-sandbox-runner,bug|improvement,severity-$sev" \
      --body "$detail + log excerpt + sandbox run-id + doobidoo hash"
    mcp__doobidoo__memory_store new problem

Issue 起票条件: severity = critical のみ Idea Issue 起票、minor は doobidoo のみ。

5.8 failure recovery (wall-clock + budget 両方で timeout)

timeout 条件:

  • wall-clock > 30 min (default、--timeout=30m)
  • budget 5h % > 50

timeout flow:

1. log 収集 (kill 前に必ず実行)
2. tmux send-keys C-c (graceful stop 試行)
3. 5 min wait
4. 未終了 → tmux kill-window
5. cleanup (worktree remove + branch delete)
6. mailbox -> admin "sandbox-timeout" mail
7. AskUserQuestion: continue / abort

implementation: admin polling で sandbox window 存在 + duration check、budget は Inv Q format で解釈
