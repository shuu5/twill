twill plugin rebuild — step verification framework

draft-v2 (2026-05-12) · LLM 責務 + 機械検証 の clean 分離 · 不変条件 U の根拠

1. 目的

新 spec の 不変条件 U (step verification post-verify 必須) を実現する共通 framework を仕様化する。各 chain step に対し「LLM が判断/作業を行い、機械が結果を検証する」という責務分離を明確化し、self-report-only な step を構造的に不能化する。

2. framing 訂正 (draft-v1 → v2)

draft-v1 の framing: 「chain-runner.sh L1873-1884 は自己申告 step、anti-pattern」と記述。LLM 実行を信用する設計自体を悪と framing していた。
draft-v2 の正しい framing (Phase 2 agent B 指摘反映):
  • chain-runner = step 記録 (record_current_step) / Skill = LLM 実行 という責務分離は妥当
  • 問題は LLM 実行を信用すること自体ではなく、「LLM 作業の結果を機械検証する layer」が不在だったこと
  • 新 spec の解決策: 責務分離を維持しつつ、step.sh framework で post-verify 機械検証を必須化

= LLM の判断責務と機械の検証責務を clean に分離。LLM は AC に従い test / 実装を書くだけ、機械は test 数 / RED→GREEN / src diff 等の客観指標で結果を verify。

3. step lifecycle (4 phase per step)

図 1: step lifecycle — 1 step につき pre-check / exec / post-verify / report の 4 phase。post-verify fail が step level fail-fast の核

4. 共通 framework: plugins/twl/scripts/step.sh

worker SKILL.md は 1 行 (step_run test-scaffold issue=$ISSUE file=$FILE) で呼び出すだけで lifecycle が回る。pseudo-bash 実装:

# worker SKILL.md は以下のように 1 行で step を呼ぶだけ
#   step_run test-scaffold issue=$ISSUE file=$FILE
#   step_run green-impl    issue=$ISSUE file=$FILE
#   step_run check         issue=$ISSUE

set -euo pipefail

# 各 step の verification rule (case で dispatch)
step_run() {
  local step_name="$1"; shift
  local args=("$@")

  # parse key=value pairs
  declare -A opts
  for arg in "${args[@]}"; do opts["${arg%%=*}"]="${arg#*=}"; done

  echo "[step.sh] ===== step=$step_name BEGIN =====" >&2

  # 1. pre-check (state snapshot)
  local pre_snapshot=$(_step_pre_check "$step_name" opts) || _step_fail "$step_name" pre "$pre_snapshot"

  # 2. exec (LLM は呼び出し元の skill 内で実行済み、step.sh は LLM 作業の verify のみ)
  # (step_run は post-verify と report のみを請け負う)

  # 3. post-verify (step 固有の rule)
  local verify_result=$(_step_post_verify "$step_name" "$pre_snapshot" opts) || \
    _step_fail "$step_name" post "$verify_result"

  # 4. report (pilot mailbox に成功 mail を書く)
  _step_report "$step_name" success "$verify_result" opts
  echo "[step.sh] ===== step=$step_name OK =====" >&2
}

# step 固有 rule の dispatch
_step_post_verify() {
  local step="$1"; local pre="$2"; local -n opts_ref="$3"
  case "$step" in
    test-scaffold)  _verify_red_test_added "$pre" "${opts_ref[file]}" ;;
    green-impl)     _verify_src_changed_and_green "$pre" "${opts_ref[file]}" ;;
    check)          _verify_lint_typecheck "${opts_ref[file]}" ;;
    pr-test)        _verify_pr_test_pass "${opts_ref[issue]}" ;;
    ac-verify)      _verify_ac_checkbox_flip "${opts_ref[issue]}" ;;
    phase-review)   _verify_agent_return_value "${opts_ref[issue]}" ;;
    *) echo "ERROR: unknown step=$step" >&2; return 1 ;;
  esac
}

# 個別 verification 関数 (rule per step)
_verify_red_test_added() {
  local pre="$1"; local file="$2"
  local pre_count=$(echo "$pre" | jq -r '.test_count')
  local post_count=$(pytest --collect-only -q 2>&1 | tail -1 | awk '{print $1}')
  [ "$post_count" -gt "$pre_count" ] || { echo "no new test"; return 1; }
  pytest -x --tb=no -q 2>&1 | tail -1 | grep -qv "passed" || { echo "test not RED"; return 1; }
  echo "{\"red_count\": $((post_count - pre_count))}"
}

_verify_src_changed_and_green() {
  local pre="$1"; local file="$2"
  [ "$(git diff --stat src/ | wc -l)" -gt 0 ] || { echo "no src/ change"; return 1; }
  pytest -x --tb=short 2>&1 >/dev/null || { echo "test still RED"; return 1; }
  echo "{\"green\": true}"
}

_step_report() {
  local step="$1"; local status="$2"; local detail="$3"
  local pilot_mbox="$REPO_ROOT/.mailbox/${PILOT_NAME:-pilot-unknown}/inbox.jsonl"
  (
    flock -x 200
    jq -nc --arg from "${WORKER_NAME:-worker-unknown}" --arg ts "$(date -u +%FT%TZ)" \
      --arg step "$step" --arg status "$status" --argjson detail "$detail" \
      '{from:$from, ts:$ts, step:$step, status:$status, detail:$detail}' \
      >> "$pilot_mbox"
  ) 200>"$pilot_mbox.lock"
}

_step_fail() {
  local step="$1"; local phase="$2"; local reason="$3"
  _step_report "$step" "failure" "{\"phase\":\"$phase\",\"reason\":\"$reason\"}"
  echo "[step.sh] FAIL step=$step phase=$phase reason=$reason" >&2
  exit 1
}

6. worker SKILL.md での使用例 (worker-test-ready)

# worker-test-ready SKILL.md (~30 lines、機械検証は step.sh 任せ)

## 役割
Issue #N の単一 file について test-scaffold + green-impl + check を実施する。

## 入力
- $ISSUE: Issue number
- $FILE:  対象 source file path
- $PILOT_NAME: pilot session name (mailbox の宛先)
- $WORKER_NAME: 自分の name

## 処理 (LLM がこの順序で実行)

### Step 1: test-scaffold
1. Issue body の AC を読む
2. AC に対応する RED test を $FILE 用 test file に書く
3. **step verification を呼ぶ**: `step_run test-scaffold issue=$ISSUE file=$FILE`
   → step.sh が RED test 追加 + fail を検証、fail なら自動 abort

### Step 2: green-impl
1. RED test を PASS させる実装を $FILE に書く
2. **step verification を呼ぶ**: `step_run green-impl issue=$ISSUE file=$FILE`
   → step.sh が src/ 変更 + pytest 全 PASS を検証

### Step 3: check
1. linter / type-checker を実行
2. **step verification を呼ぶ**: `step_run check issue=$ISSUE file=$FILE`

## 完了報告
全 step が OK で終了したら、自動的に pilot mailbox に成功 mail が積まれる。
worker は exit 0 で終了。
worker SKILL.md は ~30 lines (現状 workflow-test-ready 数百行から大幅減)。LLM は AC に従い test / 実装を書くだけ、機械検証は step.sh が一手に引き受ける。

8. 不変条件 U との関係

本 framework は invariant-fate-table.md の新規 不変条件 U (step verification post-verify 必須) の実装。invariant 定義:

  • 全 chain step は pre-check → exec → post-verify → report の 4 phase lifecycle を MUST
  • post-verify で機械検証 (test 数増加 / RED→GREEN / src diff / Agent return value 等) を 必ず実施
  • self-report-only (record_current_step + ok のみ) は禁止
  • post-verify FAIL → step abort + pilot escalate (mail に failure 明記)

検証方法: step-verification-post-verify.bats (新規) で「test なしで green-impl 呼び出すと step abort」を verify。9 P0 bug の Bug #973 (RED merge silent rot) を構造的に不能化。
