§1. 公式 hook schema (verified、draft-v1 から訂正)

draft-v1 の誤り (verified、本 session で発覚):
  • "matchers" (複数形) は 誤 → 公式は "matcher" (単数)
  • {{tool.skill_name}} / {{matcher.expected_status}} の template 変数展開は 非サポート。すべての context は stdin JSON で渡される
  • exit 1 = deny は 誤。exit 1 は non-blocking error、deny は exit 2 または exit 0 + JSON 出力
  • decision: "approve" / "block" は 古い enum → 公式は permissionDecision: "allow"|"deny"|"ask"|"defer"

1.3 if field の syntax (verified、permission rule)

if field は permission rule syntax を使う、matcher の追加絞り込み。tool 系 event でのみ有効 (PreToolUse / PostToolUse / PostToolUseFailure / PermissionRequest / PermissionDenied)。

  • "Bash(git *)" — Bash の git subcommand
  • "Edit(*.ts)" — TypeScript file の Edit
  • "Skill(phase-impl-*)" — phase-impl で始まる skill 全件

§3. deny/allow signal (公式 verified)

Exit Code意味tool 実行
0success。stdout を JSON として parse。JSON 無し or permissionDecision: "allow" → 許可。"deny" / "ask" / "defer" → それぞれの挙動JSON 次第
2blocking error。stderr が Claude に表示、tool call は denydenied
その他 (1, 3...)non-blocking error。tool call は allow されるが error 記録allowed

3.1 JSON 出力 schema (exit 0 時、公式 verified)

{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow|deny|ask|defer",
    "permissionDecisionReason": "Human-readable reason",
    "additionalContext": "Optional context for Claude",
    "updatedInput": { "command": "modified command" }
  }
}

3.2 deny の最 restrictive 勝ち (公式 verified)

同 event の全 hook は parallel 実行、permissionDecision は deny > ask > allow の最 restrictive が勝つ。

重要: PreToolUse は permission mode より 先に発火、bypassPermissions や --dangerously-skip-permissions でも deny は効く。これが Inv W の構造的保証。

§4. tier 1 phase-gate.sh 実装 (~40 行、verified pattern)

#!/usr/bin/env bash
# tier 1: 粗フィルタ + fast path (50ms 以内、stdin JSON 受信)
# 公式 hook schema 準拠 (matcher 単数、exit 0 + JSON 出力 で deny)

set -euo pipefail

# === stdin JSON から必要 field を抽出 ===
INPUT_JSON=$(cat /dev/stdin)
SKILL_NAME=$(echo "$INPUT_JSON" | jq -r '.tool_input.skill // ""')
SESSION_ID=$(echo "$INPUT_JSON" | jq -r '.session_id // ""')
TOOL_NAME=$(echo "$INPUT_JSON" | jq -r '.tool_name // ""')

# === 粗フィルタ: phase-* 以外は素通り (Skill 以外の tool は早期 return) ===
if [[ "$TOOL_NAME" != "Skill" ]] || [[ "$SKILL_NAME" != phase-* ]]; then
  exit 0   # allow (JSON なし)
fi

# === bypass policy: user 手動指定時のみ allow ===
if [[ -n "${TWL_PHASE_GATE_BYPASS:-}" ]]; then
  jq -n --arg reason "user-bypass via TWL_PHASE_GATE_BYPASS" '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "allow",
      permissionDecisionReason: $reason
    }
  }'
  exit 0
fi

# === 詳細判定は tier 2 (mcp_tool hook) に委譲、ここでは pass ===
exit 0  # tier 2 が deny を返した場合、最 restrictive 勝ちで全体 deny

§5. tier 2 twl_phase_gate_check 実装 (MCP shadow、verified pattern)

5.1 handler 関数 (純 Python、in-process テスト可能)

# cli/twl/src/twl/mcp_server/tools.py に追加
def twl_phase_gate_check_handler(args: dict) -> dict:
    """tier 2: stateful 判定 (Project Board status query)

    Args:
        expected_status: "Idea"|"Explored"|"Refined"|"Implementing"
        skill_name: phase-impl-1660 等

    Returns:
        hookSpecificOutput を含む dict
    """
    expected_status = args.get("expected_status")
    skill_name = args.get("skill_name", "")

    # phase-N から issue 番号を抽出 (e.g. "phase-impl-1660" → 1660)
    issue_num = _extract_issue_num(skill_name)
    if not issue_num:
        # issue 番号特定不能 → allow (gate 対象外、log のみ)
        return _allow("gate skipped: no issue number")

    # Project Board status query (gh CLI 経由、キャッシュあり)
    current_status = _get_project_status(issue_num)

    if current_status != expected_status:
        return {
            "hookSpecificOutput": {
                "hookEventName": "PreToolUse",
                "permissionDecision": "deny",
                "permissionDecisionReason": (
                    f"Phase gate: skill '{skill_name}' requires status "
                    f"'{expected_status}', but Project Board shows "
                    f"'{current_status}' for issue #{issue_num}. "
                    f"Transition status manually or set TWL_PHASE_GATE_BYPASS=1."
                ),
                "additionalContext": (
                    f"Inv W enforcement: status mismatch denied phase invocation. "
                    f"Mail will be sent to administrator mailbox."
                )
            }
        }
    return _allow(f"status {current_status} matches expected")

# MCP wrapper (薄い、annotations は advisory hint)
if mcp is not None:
    @mcp.tool(
        annotations={
            "title": "Phase Gate Status Check",
            "readOnlyHint": True,
            "destructiveHint": False,
            "idempotentHint": True,
            "openWorldHint": False
        }
    )
    def twl_phase_gate_check(args: dict) -> str:
        return json.dumps(twl_phase_gate_check_handler(args))

5.2 _get_project_status helper (Project Board GraphQL or cache)

def _get_project_status(issue_num: int) -> str:
    """gh project item-list でキャッシュ、5 min TTL"""
    cache = _load_cache()
    if cache.get(issue_num, {}).get("ttl") > time.time():
        return cache[issue_num]["status"]

    # gh project field-list で field_id を 1 回キャッシュ
    proj_num = _resolve_project_number()
    result = subprocess.run(
        ["gh", "project", "item-list", str(proj_num),
         "--owner", "shuu5", "--format", "json"],
        capture_output=True, check=True
    )
    items = json.loads(result.stdout)["items"]
    for item in items:
        if item.get("content", {}).get("number") == issue_num:
            status = item.get("status", "Unknown")
            _save_cache(issue_num, status)
            return status
    return "NotInBoard"

§7. deny 経路 (administrator notify)

tier 2 が deny を返した場合、phase-gate.sh が並列で mailbox::emit で administrator に notify mail を送る:

# phase-gate.sh の deny 路径
if [[ "$DENY_REASON" != "" ]]; then
  # administrator mailbox に notify mail を書き込み
  echo "{\"from\":\"phase-gate\",\"ts\":\"$(date -Iseconds)\",\"event\":\"phase-gate-denied\",\"detail\":{\"skill\":\"$SKILL_NAME\",\"issue\":$ISSUE_NUM,\"expected\":\"$EXPECTED\",\"actual\":\"$CURRENT\"}}" \
    | flock -x .mailbox/administrator/inbox.jsonl cat >> .mailbox/administrator/inbox.jsonl
fi

§8. bypass policy

  • 許可: TWL_PHASE_GATE_BYPASS=1 env (user 手動指定、debug 用)
  • 許可: 該当 Issue の Project Board status を user が手動で変更 (root cause 解消)
  • 禁止: code 内の SKIP_*_REASON や SKIP_PILOT_GATE 等の env var による bypass — これが Bug #1660/1662/1663 の根本問題

§9. retry path (前 phase 完了待ち or user 介入)

  1. deny 受信 → administrator が mail を読み、状況を user に報告
  2. user 判断:
    • 前 phase が未完了 → 該当 phase を先に実行
    • status 遷移漏れ → user 手動で Project Board status 更新
    • 緊急 → TWL_PHASE_GATE_BYPASS=1 で bypass
  3. administrator polling cycle で status 変更を検知 → 再度 phase invocation

§10. Bug #1660/1662/1663/1684 の構造的不能化

Bug既存 root cause新 spec での不能化
#1660 SKIP_*_REASON sanitize bash env で bypass、子プロセスへ自動継承、sanitize なし bypass env 全廃。PreToolUse hook + Project Board status SSoT が唯一の gate、env による緊急 bypass のみ TWL_PHASE_GATE_BYPASS で明示
#1662 :- syntax bash の ${VAR:-default} と ${VAR-default} の意味差で env override 無効化 env override 不要。phase-gate.sh は env を読まず、tier 2 が Project Board 直接 query
#1663 env override 競合 controller_count=0 短絡 path で env reassign、override 上書き env reassign の概念廃止。Python 関数引数として override (expected_status / skill_name) を渡す
#1684 IS_AUTOPILOT cwd-guard 「autopilot mode」と「正しい worktree」を 1 env で表現したガード漏れ 独立 invariant の別 enforce。tier 1 で skill 名フィルタ、tier 2 で status query、worktree 判定は別 hook (worktree-boundary.sh)

#!/usr/bin/env bats
# Inv W enforcement test (新規 bats)

@test "phase-impl on Refined Issue is allowed" {
  # setup: Issue #1660 を Refined にする
  mock_project_status 1660 "Refined"

  # exec

    | bash plugins/twl/scripts/phase-gate.sh

  [ "$status" -eq 0 ]
  # JSON 出力なし (allow)、または "allow" を含む
}

@test "phase-impl on Todo Issue is denied (Inv W)" {
  # setup
  mock_project_status 1660 "Todo"

  # exec
  output=$(echo '{"tool_name":"Skill","tool_input":{"skill":"phase-impl-1660"}}' \
    | bash plugins/twl/scripts/phase-gate.sh)

  [ "$status" -eq 0 ]
  echo "$output" | jq -e '.hookSpecificOutput.permissionDecision == "deny"'
}

@test "bypassPermissions does not bypass tier 2 deny" {
  # setup: bypassPermissions mode でも tier 2 deny は効く (Inv W 構造的保証)
  mock_project_status 1660 "Todo"

  output=$(echo '{"permission_mode":"bypassPermissions","tool_name":"Skill","tool_input":{"skill":"phase-impl-1660"}}' \
    | bash plugins/twl/scripts/phase-gate.sh)

  echo "$output" | jq -e '.hookSpecificOutput.permissionDecision == "deny"'
}

@test "TWL_PHASE_GATE_BYPASS env allows phase-impl" {
  export TWL_PHASE_GATE_BYPASS=1
  mock_project_status 1660 "Todo"

  output=$(echo '{"tool_name":"Skill","tool_input":{"skill":"phase-impl-1660"}}' \
    | bash plugins/twl/scripts/phase-gate.sh)

  echo "$output" | jq -e '.hookSpecificOutput.permissionDecision == "allow"'
}
