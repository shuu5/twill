1. 目的

既存 twill plugin の構造的欠陥 3 種 (自己申告 step / chain SSoT 三重化 / 巨大化) を 不能化 する設計に置き換える。greenfield rewrite (2-4 ヶ月) を回避し、Strangler Fig で約 1-2 週で完遂する。

  • 9 件の P0 bug が本セッションで発見 (詳細 failure-analysis.md)
  • 横断要因 F-1 (並列 Wave 後付け) / F-2 (env var 限界) / F-3 (deploy/verify 分離欠如)

2. 新原則 (7 条)

  1. LLM は判断のために使う
  2. 機械的処理は Claude Code primitives (hook / Skill / Agent / MCP の連携) で実現
  3. 自作 script / 自作 MCP server は最小限 (validate / status query の thin helper のみ)
  4. administrator (1 個、user 代理) が全体采配、Project Board status を SSoT として参照
  5. phase-* controller は administrator が 1 window = 1 個ずつ spawn (context reset、純粋指令)
  6. status と phase は 1:1 対応 (各 phase は 1 status 遷移のみ担当)
  7. status gate は PreToolUse hook で機械化 (前 status でない phase invoke は deny)

3. 3 階層 architecture (図 1)

図 1: architecture overview — 3 階層 pyramid (administrator / pilot / worker)。各層は tmux 経由で別 main session として起動、上位層は下位 mail のみで context 効率化

4. Key Points (3 階層 pyramid)

  • L0 administrator = 1 個、長命 (旧 su-observer 相当)、user 代理。pilot からの集約 mail のみ受信、context 効率化を保つ
  • L1 phase-* (pilot) = サブリーダー、各 status 遷移を 1 個ずつ担当、worker 群を tmux で spawn・管理、mail を集約して administrator に上申
  • L2 worker-* = 実行部隊、各 worker は別 main session、内部で optional に Agent subagent 1 階層 (公式制約: subagent は子 subagent spawn 不可)、結果を pilot mailbox に write
  • tool-* = user / admin が直接 invoke、status 遷移と無関係 (architect / project / utility / self-improve)
  • spawn 機構 = 全層 tmux new-window + claude code 別 main session (subagent 制約は worker 内部のみ)
  • 通信機構 = 階層型 file mailbox (.mailbox/<session-name>/inbox.jsonl + flock atomic)、pyramid 型集約
  • status SSoT = GitHub Project Board の Status field (6-stage: Idea/Explored/Refined/Implementing/PR Reviewed/Merged、Verified 削除確定)
