§2. actor scope (Phase I dig Q1 確定)

Monitor primitive 使用権限を持つ actor: administrator (L0) + phaser (L1) のみ。specialist / workflow / atomic / tool-* は使用不可。

administrator
主用途。.mailbox/administrator/inbox.jsonl + budget status line + tmux state 集約。monitors.json daemon を所有。
phaser
atomic spawn 後の .mailbox/<phaser-name>/inbox.jsonl tail で atomic 進捗監視。制約: Monitor token 二重消費を避けるため、admin と同一 channel を tail しない (channel 分離: admin = phaser 集約 mail / phaser = atomic mail)。phaser invocation 終了時に Monitor task も stop (lifecycle 一致)。
specialist / workflow / atomic / tool-*
使用禁止。run-to-completion 型のため Monitor 不要、blocking exit code + mail emit で十分。

§3. signal primary strategy (Phase I dig Q2 確定)

Monitor pattern の signal source は mailbox JSON event 単一化。旧 .supervisor/events/ ファイル方式 (existence / mtime / JSON body 混在) は 第 5 弾 dig Round 10 で廃止済。

3.1 canonical Monitor pattern

# Monitor task 起動 (administrator 用、monitors.json entry)
tail -F .mailbox/administrator/inbox.jsonl 2>/dev/null \
  | jq -rc --unbuffered 'select(.event != null) | "[\(.event)] from=\(.from) ts=\(.ts) detail=\(.detail // {} | tojson)"'

# Monitor が match した line は administrator の notification として
# conversation context に挿入される (token 消費あり)

3.2 mail schema (glossary.html canonical)

{
  "from": "<sender-id>",          // e.g. "phaser-impl-1234"
  "to": "administrator",            // recipient
  "ts": "2026-05-15T12:34:56+09:00",
  "event": "<event-type>",          // e.g. "idle-completed" / "phase-failed"
  "detail": { ... },                // event-specific structured data
  "heartbeat_ts": "..."             // optional
}

3.3 forbidden patterns (Inv T 強化)

  • .supervisor/events/<event-name>-* ファイル存在検知 (廃止済)
  • .supervisor/events/<event-name>-* mtime polling (廃止済)
  • JSON body を Monitor pattern で複雑 parse (jq は許容、Python/awk による複雑 logic は禁止)
  • 他 actor の mailbox tail (channel isolation: admin は administrator/、phaser は <own-phaser-name>/ のみ)

§4. composite event policy (Phase I dig Q3 確定)

Monitor primitive の制約 (regex 単一 / line=1 event) のため、composite condition (AND / OR / threshold / debounce) は mail emitter side で評価。Monitor は 1 mail = 1 composite event として受信。

4.1 設計原則

原則 C-1 (emitter responsibility)
composite logic (debounce / multi-indicator AND / threshold) は mail を emit する side (administrator polling cycle / phaser polling / atomic-mail-send.sh 等) で評価する。Monitor 側で複合判定しない。
原則 C-2 (one mail = one event)
composite condition が成立した場合のみ event field を持つ 1 mail を emit。複数条件が並列成立しても 1 mail に集約。
原則 C-3 (LLM evaluator は分離)
LLM-based composite 判定 (例: 自然言語 phrase classification) は administrator polling cycle (admin-cycle.html §3.4 spawn 判断 + §2 図 judge ボックス) の SKILL.md 内部で実行。Monitor pattern には埋め込まない。

4.2 例: IDLE-COMPLETED composite event

旧 cld-observe-any LLM evaluator (debounce 60s + pane content + mtime + completion phrase regex の 4 AND) を mail emitter pattern に置換:

# administrator polling cycle 内で composite check (admin-cycle.sh)
check_idle_completed() {
  local target_window=$1
  local debounce_sec=60
  local now=$(date +%s)

  # 4 indicator AND
  local pane_state=$(tmux display-message -t "$target_window" -p '#{pane_in_mode}' 2>/dev/null || echo "1")
  local last_mail_ts=$(jq -r 'select(.from == "'$target_window'") | .ts' .mailbox/administrator/inbox.jsonl | tail -1)
  local last_mail_age=$((now - $(date -d "$last_mail_ts" +%s 2>/dev/null || echo 0)))
  local phrase_match=$(tmux capture-pane -t "$target_window" -p | tail -5 | grep -cE '^✓|^完了|^Done')
  local debounce_passed=$((last_mail_age >= debounce_sec))

  if [[ "$pane_state" == "0" && "$phrase_match" -ge 1 && "$debounce_passed" == "1" ]]; then
    # composite condition 成立 → 1 mail emit (Phase 6 review CRITICAL-2 fix: mailbox_emit 関数を使用)
    source plugins/twl/scripts/lib/mailbox.sh
    mailbox_emit "$target_window" "administrator" "idle-completed" \
      "$(jq -nc '{debounce_passed: true, pane_state: "idle", phrase_matched: true}')"
  fi
}

§5. Monitor daemon lifecycle (Phase I dig Q4 確定)

5.1 plugin-managed daemon (canonical)

primary 方式: plugins/twl/monitors.json entry で Plugin loader が plugin load 時に background process を自動起動・cleanup する。

// plugins/twl/monitors.json (Phase 1 PoC で新規作成)
{
  "monitors": [
    {
      "id": "admin-inbox-tail",
      "description": "administrator mailbox event stream",
      "command": "tail -F .mailbox/administrator/inbox.jsonl | jq -rc --unbuffered 'select(.event != null) | \"[\\(.event)] \\(.detail // {} | tojson)\"'",
      "timeout_sec": 3600,
      "stderr_sink": ".supervisor/monitor-stderr-admin-inbox.log",
      "auto_restart": true
    }
  ]
}

5.2 administrator Step 0 health-check

plugin loader が起動失敗した場合に備え、administrator SKILL.md Step 0 で daemon 生存を確認:

# administrator SKILL.md Step 0 (session 起動時) — Monitor daemon health-check
MONITOR_LIFECYCLE_FILE=".supervisor/monitor-lifecycle.json"

# 1. lifecycle file 読み込み
if [[ -f "$MONITOR_LIFECYCLE_FILE" ]]; then
  ADMIN_INBOX_PID=$(jq -r '.daemon_pids."admin-inbox-tail" // empty' "$MONITOR_LIFECYCLE_FILE")

  # 2. 生存確認
  if [[ -n "$ADMIN_INBOX_PID" ]] && kill -0 "$ADMIN_INBOX_PID" 2>/dev/null; then
    echo "[MONITOR-HEALTH] admin-inbox-tail PID=$ADMIN_INBOX_PID alive"
  else
    # 3. crash 検知 → user escalate (administrator 自己 restart は循環依存)
    bash plugins/twl/scripts/record-detection-gap.sh \
      --type missing-monitor \
      --detail "admin-inbox-tail daemon not found (PID=$ADMIN_INBOX_PID)" \
      --severity high
  fi
fi
