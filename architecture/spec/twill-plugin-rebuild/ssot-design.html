§2. 3 案 trade-off

案 1: deps.yaml SSoT (chain.py + chain-steps.sh を削除、deps.yaml から生成)

利点欠点
LLM 可読性最高 (YAML、宣言的)
CRG の依存可視化との整合 (deps.yaml は CRG 入力)
step リスト + metadata を 1 箇所で表現
Python/Bash からの型安全な参照が困難 (YAML parser 必須)
runtime での next-step 計算を deps.yaml parser に委ねる必要
YAML を Python で parse してイテレートする処理が分散する
step 実装 (bash) は別 file、deps.yaml とは別管理になる (file 削減効果薄)

案 2: chain.py SSoT (現状維持・強化、export 自動化)

利点欠点
export API 既存 (export_deps_chains / export_chain_steps_sh)
型安全、AST parse で integrity 検証可能
chain.py 内に metadata を集約 (STEP_TO_WORKFLOW / CHAIN_STEP_DISPATCH 等)
chain-steps.sh の export が手動 (CI で自動 export + コミットの仕組みが未完、pre-commit hook はオプショナル)
LLM が chain.py を参照できない場面ではブラックボックス
chain-runner.sh の bash 内 dispatch logic は別管理 (chain.py 外の case 文)
step 実装 (bash) は別 file、chain.py とは別管理 (file 削減効果薄)

案 3: step.sh 単一 SSoT (1 step = 1 file、chain-runner.sh 廃止)

利点欠点
1 step = 1 steps/<name>.sh file = 単一 SSoT (定義 + 実装 + 検証が同 file)
case 文 bloat 解消 (chain-runner.sh 1932 lines 全廃)
MCP 化との親和性高 (step を MCP tool として expose 容易)
step verification framework (Inv U) と整合 (_verify_NAME() 関数を同 file 内に持てる)
LLM が steps/ directory を listing して chain 全体を理解可能
step 単体テスト容易 (bats で per-step テスト)
既存 chain-runner.sh (1932 lines) + chain.py + chain-steps.sh の全廃 = radical rebuild コスト大
resolve_next_workflow 等の dynamic dispatch (deps.yaml の meta_chains 参照) を step.sh + helper で再設計必要
deps.yaml は workflow metadata (chain 順序、依存関係) のみ保有する役割に再定義
移行期間の dual-stack 運用 (新 step.sh と旧 chain-runner.sh 併存) が必要

結論: 案 3 を採用

判断根拠:

  1. step-verification.html で既に 8 step の _verify_NAME rule を定義 (test-scaffold / green-impl / check / pr-test / ac-verify / phase-review / pr-cycle-report / merge-gate-check)。案 3 と完全整合。
  2. 既存 draft-v1.html L1401-1406 に「chain.py / chain-steps.sh / twl check --deps-integrity 全廃」「_verify_NAME 関数を持つ step.sh が単一 SSoT」と明示。本 session で再確認、変更なし。
  3. radical rebuild コストは案 1/2 でも同等 (chain.py + chain-steps.sh + chain-runner.sh の不整合解消は本質的問題、export 自動化だけでは F-1 横断要因解消につながらない)。
  4. 1 step = 1 file の利点 = MCP 化親和性 + step verification 整合 + LLM 可読性 + 単体テスト容易性 が他案より圧倒的に高い。

3.2 step.sh の規約 (template)

#!/bin/bash
# steps/test-scaffold.sh
# Step: test-scaffold (Inv U post-verify 必須)

set -euo pipefail
source "$(dirname "$0")/lib/step.sh"

# === metadata (file 内 SSoT) ===
STEP_NAME="test-scaffold"
STEP_WORKFLOW="test-ready"
STEP_DEPENDS=("workflow-setup")     # 前提 step
STEP_NEXT=("green-impl")            # 次 step (workflow chain)

# === pre-check (Inv U pre-check phase) ===
_pre_check_test_scaffold() {
  local issue_num="$1"
  # Refined status verify (Inv W, gate hook)
  status_board::require_status "$issue_num" "Refined"
  # worktree verify (Inv B)
  step::require_worktree
}

# === exec (実装本体) ===
_exec_test_scaffold() {
  local issue_num="$1"
  local ac_file="$2"
  # ... bash 内で AC から RED test を生成
  # ... または Agent tool を呼び出す
}

# === post-verify (Inv U post-verify phase、機械検証) ===
_verify_test_scaffold() {
  local issue_num="$1"
  # 1. test 数増加を確認
  local test_count_before=$(cat .audit/$run_id/test-count-before.txt)
  local test_count_after=$(bats --list tests/bats/ | wc -l)
  [ "$test_count_after" -gt "$test_count_before" ] || step::fail "test count not increased"

  # 2. RED 状態を確認 (test 実行 → 全 fail)
  bats tests/bats/ && step::fail "RED test should FAIL but PASSED"

  # 3. src/ への変更 0 を確認 (test-scaffold は test のみ)
  local src_diff=$(git diff --name-only HEAD | grep -c '^\(cli\|plugins\)/.*\(?:\.py\|\.sh\)$' || true)
  [ "$src_diff" -eq 0 ] || step::fail "src/ should not be modified in test-scaffold"
}

# === main (step.sh framework が呼び出す) ===
step::run \
  --name "$STEP_NAME" \
  --workflow "$STEP_WORKFLOW" \
  --depends "${STEP_DEPENDS[@]}" \
  --next "${STEP_NEXT[@]}" \
  --pre-check _pre_check_test_scaffold \
  --exec _exec_test_scaffold \
  --post-verify _verify_test_scaffold \
  "$@"

3.3 step::run framework (pseudo-code、~80 lines)

plugins/twl/scripts/steps/lib/step.sh:

#!/bin/bash
# step.sh framework — Inv U 4 phase lifecycle (pre-check → exec → post-verify → report)

step::run() {
  local name workflow depends_arr=() next_arr=()
  local pre_check_fn exec_fn post_verify_fn

  while [ $# -gt 0 ]; do
    case "$1" in
      --name) name="$2"; shift 2 ;;
      --workflow) workflow="$2"; shift 2 ;;
      --depends) shift; while [[ "$1" != --* ]] && [ $# -gt 0 ]; do depends_arr+=("$1"); shift; done ;;
      --next) shift; while [[ "$1" != --* ]] && [ $# -gt 0 ]; do next_arr+=("$1"); shift; done ;;
      --pre-check) pre_check_fn="$2"; shift 2 ;;
      --exec) exec_fn="$2"; shift 2 ;;
      --post-verify) post_verify_fn="$2"; shift 2 ;;
      --) shift; break ;;
      *) shift ;;
    esac
  done

  local issue_num="${1:?issue_num required}"; shift

  # report start
  mailbox::emit "$name" "step-started" "{\"issue\":$issue_num}"
  audit_snapshot ".audit-step-pre" "$name" || true

  # 1. pre-check (Inv U pre-check phase)
  if [ -n "$pre_check_fn" ]; then
    if ! "$pre_check_fn" "$issue_num" "$@"; then
      mailbox::emit "$name" "step-precheck-failed" "{\"issue\":$issue_num}"
      audit_snapshot ".audit-step-precheck-fail" "$name" || true
      exit 2
    fi
  fi

  # 2. exec
  local exec_status=0
  "$exec_fn" "$issue_num" "$@" || exec_status=$?

  # 3. post-verify (Inv U post-verify phase、機械検証必須)
  if [ -n "$post_verify_fn" ]; then
    if ! "$post_verify_fn" "$issue_num" "$@"; then
      mailbox::emit "$name" "step-postverify-failed" "{\"issue\":$issue_num,\"exec_status\":$exec_status}"
      audit_snapshot ".audit-step-postverify-fail" "$name" || true
      exit 3
    fi
  fi

  # 4. report
  mailbox::emit "$name" "step-completed" "{\"issue\":$issue_num,\"next\":\"${next_arr[*]}\"}"
  audit_snapshot ".audit-step-post" "$name" || true
}

step::fail() {
  echo "step::fail: $*" >&2
  exit 1
}

step::require_worktree() {
  # Inv B: source-touching step は worktree 内必須
  local cwd; cwd=$(pwd)
  case "$cwd" in
    */worktrees/*) return 0 ;;
    *) step::fail "must run in worktree, but cwd=$cwd" ;;
  esac
}

3.4 resolve_next_workflow の置換

既存 chain.py の resolve_next_workflow() は deps.yaml の meta_chains を読んで「次の workflow」を計算していた。新 architecture では:

# deps.yaml.chains の新構造 (workflow metadata のみ)
chains:
  setup:
    description: "初期化 + Project Board status 遷移"
    steps:
      - workflow-setup
      - project-board-status-update
      - crg-auto-build
      - arch-ref

    description: "RED test 作成 + post-verify"
    steps:
      - test-scaffold
      - green-impl
      - check
    next_workflow: "pr-verify"
  ...

# resolve next workflow は phase-* SKILL.md で簡素に
# 例: phase-impl.md
# ```
# 1. yq '.chains.[$current_workflow].next_workflow' deps.yaml で次 workflow 名を取得
# 2. その workflow の steps[0] を step.sh で実行
# ```

deps.yaml は workflow chain の上位構造 (どの workflow がどの順序で実行されるか) のみを保有、各 step の実装と metadata は steps/<name>.sh 内に持つ。

5.1 新 audit section 9: steps_integrity

新 architecture の SSoT 整合性検証は 2-way:

項目検証内容severity
1. steps/ file 存在 deps.yaml.chains.<workflow>.steps に存在する全 step 名 × plugins/twl/scripts/steps/<name>.sh 存在 critical (orphan reference: deps.yaml にあるが file なし)
2. deps.yaml entry 存在 plugins/twl/scripts/steps/<name>.sh 存在 × deps.yaml.chains の steps list に登録 warning (orphan file: file あるが deps.yaml に登録なし)
3. step.sh metadata 整合 step file 内の STEP_WORKFLOW = deps.yaml で属する workflow critical (cross-workflow mismatch)
4. step.sh dependency 整合 step file 内の STEP_DEPENDS ⊆ deps.yaml で先行する step warning (dependency missing in chain)
5. _verify_NAME 関数存在 step.sh 内に _verify_<name>() 関数定義 (Inv U post-verify 必須) critical (missing post-verify, RED-only merge risk)
6. step::run 呼び出し step.sh が step::run framework を呼び出す (自己申告 step 防止) critical (self-report-only step, Inv U 違反)

# cli/twl/src/twl/validation/steps_integrity.py
import re, yaml
from pathlib import Path

def audit_steps_integrity(deps_yaml_path: Path, steps_dir: Path) -> list[dict]:
    items = []
    deps = yaml.safe_load(deps_yaml_path.read_text())
    chains = deps.get('chains', {})

    # 1. deps.yaml に登録された step が steps/<name>.sh として存在するか
    referenced_steps = set()
    for wf_name, wf in chains.items():
        for step_name in wf.get('steps', []):
            referenced_steps.add(step_name)
            step_path = steps_dir / f"{step_name}.sh"
            if not step_path.exists():
                items.append({
                    "severity": "critical",
                    "section": "steps_integrity",
                    "component": f"deps.yaml:chains.{wf_name}.steps",
                    "message": f"orphan reference: step '{step_name}' in deps.yaml but {step_path} not found"
                })

    # 2. steps/<name>.sh が deps.yaml に登録されているか
    for step_file in steps_dir.glob("*.sh"):
        step_name = step_file.stem
        if step_name in {"step", "mailbox", "status-board"}:  # lib/ exclusion
            continue
        if step_name not in referenced_steps:

                "severity": "warning",
                "section": "steps_integrity",
                "component": f"{step_file.relative_to(deps_yaml_path.parent)}",

    for step_file in steps_dir.glob("*.sh"):

    return items
