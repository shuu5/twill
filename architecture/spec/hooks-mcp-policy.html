§3. hook independence invariant (Inv H-1〜3)

Inv H-1 (hook independence)
各 hook は他 hook の中間 state に依存しない。公式仕様「Hook chain (複数 hook 同一 matcher) の実行順序は保証なし」のため、順序依存は破綻リスクあり。共有 state は file 経由のみ (.audit/<run-id>/ or .mailbox/<session>/) で交換、env 変数や process 内部 state を inter-hook で共有しない。
Inv H-2 (idempotent hook)
各 hook は idempotent (同 input → 同 output)。multiple fire でも safe、PostToolUse / PostToolUseFailure では特に重要。
Inv H-3 (hook output 形式統一)
hook output は outputType: log (debug log のみ) or outputType: decision (JSON permissionDecision) のいずれか。混在禁止。
例外 (順序依存が真に必要な場合)
単一 wrapper script (pre-bash-master.sh 等) に sequential chain を集約、spec で明示。例: pr-test → merge-gate → final-decision の連鎖が必要なら 1 script 内で順序固定。

EXP-039: 公式仕様「順序保証なし」が実装上どの程度厳密かを empirical verify。詳細 §7。

§5. PostToolUseFailure + memory_store 自動学習

5.1 設計思想

tool 実行失敗時の PostToolUseFailure hook で mcp__doobidoo__memory_store を呼び、失敗 pattern を doobidoo memory に type: feedback として自動保存する。将来同 pattern が memory_search hit したら LLM が warning 表示。

5.2 現状実装 (command hook 経由)

mcp_tool hook の template 変数展開 (${tool_input.command} 等) は公式 docs に明示なし。現状仕様: command hook で script 起動し、script が stdin JSON を parse して additionalContext で LLM に memory_store を促進する形式を採用する。

# plugins/twl/hooks/hooks.json (公式 verified 入れ子 schema)
{
  "hooks": {
    "PostToolUseFailure": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/scripts/hooks/post-tool-use-failure-learn.sh",
            "timeout": 10000
          }
        ]
      }
    ]
  }
}

# post-tool-use-failure-learn.sh 内で stdin JSON parse、additionalContext で LLM に memory_store 促進
# 実装参照: plugins/twl/scripts/hooks/post-tool-use-failure-learn.sh
# hook input field: tool_name / tool_input / error / error_type / is_interrupt / is_timeout
mcp_tool hook 直接形式 (type: "mcp_tool" + tool: "mcp__doobidoo__memory_store") は template 変数展開 (${tool_input.command} 等) が公式 docs に明示されていない。直接 migrate するには EXP-043 で template 変数展開を empirical verify する必要があり、defer item (§6) として扱う。

5.3 制約

  • doobidoo memory_search hit 時の warning 表示は LLM 側 (admin / phaser SKILL.md Step 0 で memory_search 実行)
  • doobidoo memory_store 自体の失敗は silent (循環 hook 発火回避)
  • noise 過多防止: hook 内で error string の最初 200 文字のみ取り込み (長文は MCP server 側で cull)、stderr 直接参照は不可 (hook input schema に stderr field なし)

§7. EXP 追加 (smoke 候補)

EXP-039 (hook 実行順序)
公式仕様「Hook chain 順序保証なし」の Claude Code 現実装 (Opus 4.7) における実態を empirical verify。.claude/settings.json に同 matcher (Bash) で 3 dummy hook 登録、各 hook が stderr に hook-A-fired-at-$(date +%N) emit、10 回 fire で order_stability 判定。出力: {order_stable: bool, observed_orders: [...], variance: ...}。status: inferred
EXP-040 (MCP tool + prompt handler latency / cost)
LLM-based gate 判定 (複雑な IF-THEN logic) の per-hook latency + cost を empirical 計測。ADR-040 phaser → tool-* spawn gate を test 対象として、Sonnet vs Opus の latency (5-10s) + cost ($0.01) を実機計測。status: inferred
EXP-041 (Plugin scope hook schema)
plugins/<plugin>/hooks.json が Claude Code plugin loader で supported か empirical verify。schema (matcher / type / handler) が plugin loader で誤らない confirm。plugins/twl/hooks.json を新規作成して plugin reload で動作確認。status: inferred
EXP-042 (Elicitation hook 公式存在)
Elicitation hook が現 Claude Code 実装で supported か WebFetch + 実機 fire で empirical verify。approval dialog が hook で自動応答できるなら Monitor regex 代替の clean path、token 節約効果あり。status: inferred
EXP-043 (mcp_tool template 変数展開)
type: "mcp_tool" hook で ${tool_input.command} 等の template 変数が展開されるか empirical verify。verified なら §5.2 の command hook 経路を mcp_tool 直接形式に migrate 可能。status: inferred
