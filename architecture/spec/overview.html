2. 新原則 (10 条)

  1. LLM は判断のために使う
  2. 機械的処理は Claude Code primitives (hook / Skill / Agent / MCP / Plugin Monitor の連携) で実現
  3. 自作 script / 自作 MCP server は最小限 (validate / status query の thin helper のみ)
  4. administrator (1 個、user 代理) が全体采配、Project Board status を SSoT として参照
  5. phaser-* controller は administrator が 1 window = 1 個ずつ spawn (context reset、純粋指令)
  6. status と phaser は 1:1 対応 (各 phaser は 1 status 遷移のみ担当、phaser は canonical 1 語)
  7. status gate は PreToolUse hook で機械化 (前 status でない phaser invoke は deny)
  8. twl は Claude Code 公式 plugin として正式化: .claude-plugin/plugin.json + skill namespace /twl:* + Plugin Monitor + --plugin-dir local test
  9. 各 spec claim は 4-state verification status で明示: inferred → deduced → verified → experiment-verified。experiment-id hyperlink で sandbox 実地検証へ link
  10. 10 role + vocabulary 統合 SSoT: 10 role (administrator / phaser / tool / workflow / atomic / specialist / reference / script / hook / monitor) + plugins/twl/registry.yaml 1 file が glossary + components + chains + hooks+monitors + integrity_rules の 5 section を保有、「1 entity = 1 name」ルールで twl audit --vocabulary 機械検証

3. 10 role architecture (図 1)

role の詳細 matrix: registry-schema.html §3 (10 role × concern matrix)
図 1: 10 role architecture overview — L0 administrator / L1 phaser-* + tool-* (independent) / L2-3 workflow + atomic + specialist。phaser は別 main session、workflow/atomic は Skill composition、specialist は Agent で spawn (subagent context)

4. Key Points (10 role architecture)

  • L0 administrator = 1 個、長命 (旧 su-observer rebrand)、user 代理。phaser からの集約 mail のみ受信、context 効率化を保つ
  • L1 phaser-* = サブリーダー、各 status 遷移 (status-transition) を 1 個ずつ担当 (旧 pilot/phase/controller 5 重を phaser 1 語に統一)。workflow を Skill で呼ぶ、必要なら Agent(specialist-*) も
  • L2 workflow-* = phaser/tool から Skill で呼ばれる sub-workflow、atomic を順次呼ぶ launcher (disable-model-invocation: true、context budget 外)
  • L3 atomic-* = 最小実行単位、4-phase lifecycle (pre-check/exec/post-verify/report) を SKILL.md 本文に inline 実装、必要なら Agent(specialist-*) で specialist spawn
  • specialist-* (公式 agents/ directory) = Agent tool 経由 sub-agent、独立 context window、subagent spawn 不可 (verified)。旧 worker-* 18 件を rename + 新規 specialist-spec-* 6 件 (explorer / architect / review-{vocabulary,structure,ssot,temporal}、tool-architect 7-phase PR cycle 用、tool-architecture.html §3.7)
  • tool-* = user / admin が直接 invoke、status 遷移と無関係 (tool-architect / tool-project / tool-sandbox-runner の 3 件)
  • spawn 機構 = phaser は tmux new-window + claude code 別 main session、workflow/atomic は Skill 公式機構、specialist は Agent 公式機構
  • 通信機構 = 階層型 file mailbox (4 階層 entity: mailbox > inbox > mail > event、.mailbox/<session>/inbox.jsonl + flock atomic)、pyramid 型集約
  • status SSoT = GitHub Project Board の Status field (6-stage: Idea/Explored/Refined/Implementing/PR Reviewed/Merged、forbidden synonym: state)
  • vocabulary SSoT = registry.yaml#glossary、「1 entity = 1 name」ルール、twl audit --vocabulary 機械検証
