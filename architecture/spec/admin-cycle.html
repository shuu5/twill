§1. 公式 CronCreate 制約 (verified、設計に重大影響)

重要な公式制約:
  • session-scoped only: scheduled tasks は current conversation に live、session 死亡で task も死亡。新 conversation 開始で全 task clear
  • durable: true は BROKEN (Issue #40228 OPEN): 公式 schema には記載なし、output に "Session-only" と表示、~/.claude/scheduled_tasks.json は作成されない
  • --resume/--continue で復元 (7 日以内 recurring、未 fire one-shot のみ)
  • 7 日 expiry: recurring task は creation から 7 日後に自動 expire (1 回 fire してから削除)
  • 50 task 上限 / session
  • jitter: recurring up to 30 min (or half-interval)、one-shot top/bottom hour up to 90 sec 早く
  • 1 分粒度の standard 5-field cron expression
  • tools: CronCreate / CronList / CronDelete (ScheduleWakeup は本 file context に存在せず、/loop dynamic mode で代替)
公式引用: 「Tasks are session-scoped: they live in the current conversation and stop when you start a new one. Resuming with --resume or --continue brings back any task that hasn't expired」「Recurring tasks automatically expire 7 days after creation. The task fires one final time, then deletes itself.」
source: code.claude.com/docs/en/scheduled-tasks

1.1 永続化代替 (admin が machine 再起動を生き残るには)

CronCreate は session-scoped、admin session が --resume なしで起動された場合は task 不在。永続化代替は 3 つ:

代替用途本 spec での扱い
Routines (cloud)Anthropic-managed infra、machine off でも動く、fresh clonetool-* など stateless task 向け、admin は使用不可 (local file 必要)
Desktop scheduled taskslocal machine cron、Claude Code 起動を scheduleadmin session 自動起動の手段として候補 (machine boot で起動)
GitHub Actions schedulerepo CI、cron-driven workflowcleanup / archive 系 backup task に使用

本 spec の方針: admin session は user が手動起動 + --continue で resume、CronCreate (recurring=true、durable 引数なし) で session 内 polling。machine 再起動時は user が手動 resume、または Desktop scheduled tasks で自動起動を schedule する。

§2. polling cycle 6 step (verified)

図 1: administrator polling cycle — read 3 (inbox / Board / tmux) → judge → write 3 (spawn / escalate / cleanup)

§3. 各 step の詳細 (verified pattern)

3.1 Step 1: .mailbox/administrator/inbox.jsonl 読込

各 phaser が emit した mail を読み、未読 line を line 番号で track。flock 共有 lock で read。format:

{"from":"phaser-impl-1660","ts":"<ISO8601>","event":"phase-completed","detail":{"issue":1660,"status":"Implementing"}}
{"from":"phaser-pr-1660","ts":"<ISO8601>","event":"merge-gate-failed","detail":{"issue":1660,"reason":"specialist FAIL"}}
{"from":"phase-gate","ts":"<ISO8601>","event":"phase-gate-denied","detail":{"skill":"phaser-impl-1700","issue":1700,"expected":"Refined","actual":"Todo"}}

3.2 Step 2: GitHub Project Board status poll

gh project item-list <PROJ_NUMBER> --owner <OWNER> --format json で全 Issue Status + 前回 cycle との diff。5 min cache で API rate limit 緩和。

3.3 Step 3: tmux windows 列挙

tmux list-windows -a -F '#{window_name}|#{pane_dead}' で全 phaser session の生存確認 (Inv G クラッシュ検知保証、format 変数 verified)。specialist は subagent context で動作するため tmux window 不在。

3.4 Step 4: spawn 判断

各 Issue について「次 phase を起動すべきか?」を判定。判断ロジック:

FOR each issue IN board_items:
  current_status = issue.status
  has_active_phaser = ANY tmux window prefix "phaser-*" matches issue
  next_phase = STATUS_TRANSITION_MAP[current_status]  # 例: Refined → phaser-impl

  IF current_status NOT IN ["Idea", "Explored", "Refined", "Implementing"]:
    CONTINUE  # 終端 status は spawn 不要

  IF has_active_phaser:
    CONTINUE  # Inv I-1 idempotent guard

  IF active_phaser_count >= MAX_PARALLEL_PHASERS:
    DEFER  # Inv I-2 parallelism 上限

  IF unread_phaser_mail_for_issue:
    PROCESS_MAIL_FIRST  # Inv I-3 mail check prerequisite

  spawn_tmux phaser-{next_phase}-{issue_num}

3.5 Step 5: failure / escalate

  • phaser failure mail (phase-gate-denied / phase-failed / specialist-FAIL) を集約 (mail event 名の phase-* prefix は spawn-protocol §6.1 で定義された canonical event 名、role prefix とは独立)
  • 3 回連続 failure → ★HUMAN GATE escalate (user mail or AskUserQuestion)
  • budget exhaustion mail (Inv Q) → pause + recovery (CronCreate を YYm+5min 後に reschedule)

3.6 Step 6: cleanup

  • completed phaser window (status=PR Reviewed / Merged) は tmux kill-window -t <name>
  • .mailbox/<completed>/inbox.jsonl を .mailbox/archive/inbox-<ts>.jsonl に archive
  • 7 日以上 stale の archive を削除 (disk pressure 緩和)

§4. 不変条件 I-1〜I-7 (verified、本 spec 確定)

I-1
同 Issue に 1 個のみ phaser spawn (idempotent guard)。spawn 前に tmux list-windows で確認、active なら skip。
I-2
parallelism 上限 (default 3) 超過時は defer。MAX_PARALLEL_PHASERS env で override 可能。budget 5h budget %、context 効率を考慮した default。
I-3
mail check が next spawn の prerequisite。新 phaser spawn 前に当該 Issue の未読 phaser mail を全件 process。
I-4
administrator は atomic/specialist のことを直接知らない。phaser mail のみ受信、atomic/specialist 直接 mail 不可。pyramid 集約 model (specialist → atomic → workflow → phaser → administrator)。
I-5
status 更新は phaser 任せ (admin は read のみ)。gh project item-edit 実行は phaser 専任、admin が status を更新する場合は user 明示介入時のみ。
I-6
escalate 時は user 確認。3 回連続 failure / 不明 mail / budget exhaustion → AskUserQuestion or mail to user。autonomous 判断は禁止。
I-7
tmux send-keys 使用禁止 (mailbox 経由のみ)。tmux 操作は spawn (new-window) と list (list-windows / format 変数) のみ、send-keys / capture-pane で他 session に inject 禁止 (escape 罠、cross-session pollution 防止)。

§5. CronCreate 設定 + recovery

5.1 起動時 set

administrator/SKILL.md Step 0 (session 起動時) で実施する 3 step:

stepactiontool / data source
1. 既存 task 確認 (resume 時)CronList から admin-cycle prompt prefix の task を filterbuilt-in CronList tool
2. 不在なら CronCreate setcron="*/25 * * * *" (25 分間隔、jitter ≤ 12.5 min) / prompt="run admin-cycle.sh polling iteration" / recurring=truebuilt-in CronCreate tool (durable 引数は #40228 BROKEN のため未使用、session-scoped で十分)
3. Monitor daemon health-check.supervisor/monitor-lifecycle.json から daemon_pids."admin-inbox-tail" 取得 → kill -0 $PID で生存確認 → crash 時に escalateplugins/twl/monitors.json daemon (plugin loader で auto-start) + monitor-policy.html §5.2
crash 検知時の record-detection-gap.sh escalation helper は未実装 (HIGH defer item、別 change package で扱う)。現状は user 手動 escalate で運用。

5.2 7 日 expiry 対応

recurring task は 7 日後 expire (1 回 final fire してから削除)。対策:

  • admin SKILL.md の polling iteration 内で CronList を確認
  • 残 expiry < 2 日 → 次 CronCreate を set + 当該 CronDelete (atomic substitution)
  • admin session が 7 日連続稼働しないと想定するなら、初回 set のみで十分

5.3 budget pause 対応 (Inv Q)

budget 5h cycle reset を待つ場合:

  • budget status line から (YYm) 抽出
  • CronDelete で polling 停止
  • (YYm) + 5 min 後に再 set (CronCreate one-shot で wakeup → admin が recurring を再 set)

§7. 案 2 の致命的制約 + safe degradation

制約: phaser / atomic (phaser-* / atomic-* / specialist-* / workflow-*) が未配置の段階では、admin が status 遷移を検知して spawn コマンドを発行しても spawn 先 SKILL.md が存在しないため即エラー。

safe degradation 設計:

  1. admin-cycle.sh の step 4 (spawn 判断) で「target SKILL.md が存在するか」を pre-check
  2. 存在しない場合は safe-no-op mail を mailbox に emit、admin log のみ
  3. user が phaser-* / workflow-* / atomic-* / specialist-* 実装を進めるに従い、spawn 対象が増えていく段階的 rollout

この設計により、admin 単独動作で:

  • mailbox.sh flock atomic write (Inv T) 動作確認
  • CronCreate recurring + jitter 動作確認
  • tmux list-windows format 変数 動作確認
  • gh project item-list キャッシュ 動作確認

= PoC verify points の半数を admin 先行実装で確認可能。
