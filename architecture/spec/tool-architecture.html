§3. tool-architect 詳細

3.1 役割

  • HTML 形式 architecture spec を 「現仕様」として対話的に維持する
  • feature-dev workflow 経由 (Issue を介さない直接対話)、後の実装指針の大本
  • spec の 唯一 author (本 spec 自体も tool-architect 経由で生成可能、recursive OK)

3.2 verify 強制機構 (2 層併用) EXP-027

層 1: SKILL.md MUST (LLM 規律)
  • Step 0: spec 全 file を grep して 4-state status (inferred / deduced / verified / experiment-verified) を listing、未 verified を todo 化
  • Step N: 新 feature 追加時に EXP 候補生成必須 (experiment-index.html へ追記)
層 2: PostToolUse hook verify-coverage.sh (機械検証)

新規 helper、plugins/twl/scripts/lib/verify-coverage.sh (~50 行)。動作:

  1. grep -c '<span class="vs inferred">' spec/*.html → N inferred (warn)
  2. grep -c '<span class="vs deduced">' spec/*.html → M deduced (warn)
  3. git diff HEAD -- spec/*.html で new section に <a class="exp-link"> 不在を warn
  4. changelog.html を git diff と突合、本日 entry 不在を warn

振る舞い: warn のみ (block せず)、PR review で specialist-spec-review-* 4 並列が評価して fix 要求 (§3.7.3 参照)。

2 層必須の根拠 (dig core lesson §5): LLM だけ (層 1) では Step 0 漏れで verify gap、hook だけ (層 2) では context 欠如で false positive。両層併用で漏れと過剰検出の両方を抑制。

本 §3.2.2 は §3.2 の 2 層 verify を発展させた L1〜L5 5 層防御。change 001-spec-purify (2026-05-16) で確定。各層は独立に検証 + 上位層への escalate を担い、単層 bypass による drift を防ぐ。

多層化の根拠: 単層 (LLM 規律 or hook のみ) では bypass 容易。L1 (規律) を L2 (test) で reify、L3 (hook+MCP) で runtime enforce、L4 (pre-commit) で local guard、L5 (CI) で remote guard。各層は次層への escalation 経路を保つ。warning-only 方針: false positive 許容のため findings > 0 でも block しない (CRITICAL 違反のみ後続 step で fail)。

L4 defer 根拠: pre-commit hook の Vale + textlint 機械化は HIGH defer task として独立する (ADR-0013 §References 参照)。L5 CI のみで mechanization を完結し、ローカル開発時は L1 規律 + L3 hook + L3 MCP の 3 重 guard で代替する。

3.3 7-phase multi-agent PR cycle (feature-dev 型 transposition) EXP-029

3.8 scope-based fix loop SLA

Phase F review findings の自己 fix loop 上限は edit scope に応じて動的に決定。

scope判定基準fix loop 上限escalation
small1 file 単独 / edit < 20 行 / structural change なし3 回3 回到達 → user AskUserQuestion approve
medium2-5 file / edit 20-100 行 / structural change 小5 回5 回到達 → user AskUserQuestion approve + 残 findings listing
large6+ file / edit 100+ 行 / R-10 dir 構造変更 / 新 file 追加user 個別判断 (各 review round 後に AskUserQuestion)進捗 ≥ 80% で merge or revert 提案

scope 判定の根拠: large scope (8 file / 100+ 行 / structural change あり) では固定 3 回 fix loop では深部 finding に到達せず、5 回でも一部漏れる observation がある。動的判定で適切な loop 量を確保する。

recursive PR の特殊扱い: tool-architect が自身の spec (本 file 等) を更新する場合、Phase F は user / 他 tool に委譲 (self-review bias 回避)。fix loop は user 主導で実施。

5.7 doobidoo 重複 check + Idea Issue 起票 EXP-031
