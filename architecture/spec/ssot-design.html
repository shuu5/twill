§2. (2026-05-13) での再認識 (vocabulary SSoT 不在も判明)

(2026-05-13) (2026-05-13、10 round × 33+ question) で chain SSoT 三重化に加えて以下の SSoT 多重化が判明:

  • types.yaml 矛盾: cli/twl/types.yaml が component 型 + token 閾値 SSoT として記述されているが、plugin component 構成では旧 type 名 (controller / co-autopilot 等) が残存。新 type 名 (administrator / phaser-* / workflow-* / atomic-* / specialist-*) との整合性が drift。(2026-05-13) で types.yaml 自体が廃止確定、registry.yaml 統合 SSoT に移行
  • vocabulary SSoT 不在: pilot/phase/controller の 5 重、worker の 5 重、step の 5 重等の用語多重化が「1 entity = 1 name」違反、SSoT として機能不全
  • 役割名 (role) の階層分類不在: tool / pilot / worker / specialist 等の役割が 11 種以上に膨らんでいたが、階層関係 (L0/L1/L2/independent) や concern 分離が暗黙
  • 制御系 4 語多重: administrator / orchestrator / controller / supervisor の 4 語が同概念に対し並存

§3. 旧 3 案 trade-off (過去 dig、歴史的記録)

過去 dig (2026-05-12) で検討された 3 案を歴史的記録として保全。(2026-05-13) で「案 4 registry.yaml 統合 SSoT」に進化したため、いずれの旧案も廃案だが、判断経緯として残す。

案 1 (廃案): deps.yaml SSoT (chain.py + chain-steps.sh 全廃、deps.yaml から生成)

YAML editing は人間にとって直感的、yamllint integration、コメント sup
deps.yaml v3.0 既存巨大化 (~250 行)、bash dispatch logic を yaml 内 inline は無理、chain.py を捨てると Python validation も再実装が必要

案 2 (廃案): chain.py SSoT (chain-steps.sh + deps.yaml.chains 全廃、Python が SSoT)

案 3 (廃案、過去 dig 採用 → (2026-05-13) で廃案化): step.sh 単一 SSoT

1 step = 1 steps/<name>.sh file = SSoT、 _verify_<name> 関数で post-verify rule 同 file 内に、最も「分散 SSoT」原則に忠実
(2026-05-13) 発覚問題: (1) step は forbidden 一般語、atomic が canonical、(2) step.sh framework の step::run bash 呼び出しは公式 skill 機構 (Skill tool + atomic SKILL.md) と二重 framework になる、(3) vocabulary / types.yaml と独立した別 SSoT を作るため SSoT 多重化解消にならない

§5. atomic skill composition pattern ((2026-05-13) Round 6 確定)

詳細仕様: atomic-verification.html (atomic skill 4-phase lifecycle specification、旧 step-verification.html から rename)

5.1 呼び出し chain (公式仕様 verified)

[user / administrator が直接 invoke]
    │ slash command
    ▼
[administrator skill] ← description あり
    │ tmux new-window + claude --plugin-dir + slash command
    ▼
[phaser-impl skill] ← 別 session window、description あり
    │ Skill(workflow-test-ready)
    ▼
[workflow-test-ready skill] ← disable-model-invocation: true
    │ Skill(atomic-test-scaffold) → Skill(atomic-green-impl) → ...
    ▼
[atomic-test-scaffold skill] ← disable-model-invocation: true、4-phase lifecycle inline
    │ Agent(specialist-prompt-reviewer) ← 必要時のみ
    ▼
[specialist-prompt-reviewer agent] ← Agent tool、subagent context

5.2 atomic SKILL.md inline 4-phase lifecycle

旧 step.sh framework の外部 bash 呼び出しを廃止し、atomic SKILL.md 本文に Step 0〜Step 3 として 4-phase を inline 実装。Step N は公式 SKILL.md スタイルの手順番号 (backtick + 大文字、migration-stage entity との混同なし)。

---
disable-model-invocation: true
allowed-tools: [Bash, Read, Edit, Agent]
---

# atomic-test-scaffold

`Step 0` lifecycle phase: pre-check
- !`bats --list tests/ | wc -l > /tmp/pre_test_count`
- !`git diff --name-only HEAD | grep '^src/' | wc -l > /tmp/pre_src_diff`
- Verify `status` == Refined via !`gh project item-list ...`

`Step 1` lifecycle phase: exec
- Read $ARGUMENTS[0] (AC file for Issue)
- Generate RED test targeting the AC, Write to tests/unit/...

`Step 2` lifecycle phase: post-verify
- !`POST=$(bats --list tests/ | wc -l); PRE=$(cat /tmp/pre_test_count); [ "$POST" -gt "$PRE" ] || exit 1`
- !`bats tests/unit/... 2>&1 | grep -q "not ok"` # RED 確認
- !`cat /tmp/pre_src_diff | grep -q '^0$'` # src/ 変更 0 確認

`Step 3` lifecycle phase: report
- !`jq -nc '{from:"atomic-test-scaffold",ts:"'"$(date -u +%FT%TZ)"'",event:"step-completed",detail:{...}}' >> .mailbox/$ARGUMENTS[session]/inbox.jsonl`
