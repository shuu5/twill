status: done · atomic skill composition + vocabulary table 統合
目的: 既存 twill plugin の chain 定義 SSoT が chain.py / chain-steps.sh / deps.yaml.chains の三重化状態にあり (ADR-022 で正式定義)、Issue #853 で integrity 違反が顕在化。さらに cli/twl/types.yaml (component 型 + token 閾値 SSoT) と命名 vocabulary SSoT 不在の課題も併存する。本設計で 1 つの統合 SSoT に絞る。

旧 chain.py + chain-steps.sh + chain-runner.sh + cli/twl/types.yaml + 過去 dig step.sh framework を全廃。

§2. chain SSoT 以外の多重化課題

chain SSoT 三重化に加えて以下の SSoT 多重化が併存する:

  • types.yaml 矛盾: cli/twl/types.yaml が component 型 + token 閾値 SSoT として記述されているが、plugin component 構成では旧 type 名 (controller / co-autopilot 等) が残存し、新 type 名 (administrator / phaser-* / workflow-* / atomic-* / specialist-*) との整合性が drift する。types.yaml は廃止し registry.yaml 統合 SSoT に移行する
  • vocabulary SSoT 不在: pilot/phase/controller の 5 重、worker の 5 重、step の 5 重等の用語多重化が「1 entity = 1 name」違反、SSoT として機能不全
  • 役割名 (role) の階層分類不在: tool / pilot / worker / specialist 等の役割が 11 種以上に膨らんでいたが、階層関係 (L0/L1/L2/independent) や concern 分離が暗黙
  • 制御系 4 語多重: administrator / orchestrator / controller / supervisor の 4 語が同概念に対し並存

§3. 採用案 (registry.yaml 統合 SSoT)

§5. atomic skill composition pattern

詳細仕様: atomic-verification.html (atomic skill 4-phase lifecycle specification)

5.1 呼び出し chain (公式仕様 verified)

[user / administrator が直接 invoke]
    │ slash command
    ▼
[administrator skill] ← description あり
    │ tmux new-window + claude --plugin-dir + slash command
    ▼
[phaser-impl skill] ← 別 session window、description あり
    │ Skill(workflow-test-ready)
    ▼
[workflow-test-ready skill] ← disable-model-invocation: true (Skill tool は使える、inferred、EXP-010 で smoke 検証予定)
    │ Skill(atomic-test-scaffold) → Skill(atomic-green-impl) → ...
    ▼
[atomic-test-scaffold skill] ← disable-model-invocation: true、4-phase lifecycle inline
    │ Agent(specialist-prompt-reviewer) ← 必要時のみ
    ▼
[specialist-prompt-reviewer agent] ← Agent tool、subagent context

5.2 atomic SKILL.md inline 4-phase lifecycle

外部 bash 関数 (step::run) を採用せず、atomic SKILL.md 本文に Step 0〜Step 3 として 4-phase を inline 実装する。Step N は公式 SKILL.md スタイルの手順番号 (backtick + 大文字、migration-stage entity との混同なし)。具体的な inline 実装例 (Step 0 pre-check / Step 1 exec / Step 2 post-verify / Step 3 report) および disable-model-invocation: true + allowed-tools: [Bash, Read, Edit, Agent] frontmatter の rationale は atomic-verification.html §3 を参照する。mailbox identifier 注入規約 ($TWL_PHASER_NAME env 経由、Step 3 で .mailbox/$TWL_PHASER_NAME/inbox.jsonl 参照) は atomic-verification.html §3.1 が SSoT。
