3.2 動作仕様 (~50 行 helper の責務)

spawn-tmux.sh は 3 階層共通 spawn helper。tmux new-window で detached session を生成し、claude code を起動した後、slash command を inject して skill 呼び出しを trigger する。verified flags: tmux new-window -d/-n/-c/-e (tmux(1))。

step動作verify rule
1. 引数 parse5 引数を while/case で取得 (--window-name / --skill / --args / --cwd / --mailbox-from)必須引数欠落で exit 1
2. window 名 length checktmux 制約に従い 50 文字上限を assert50 文字超で _fail
3. idempotent guard (Inv I-1)tmux list-windows -a -F '#{window_name}' | grep -qx で既存 window 検査既存なら spawn-skip mail emit して return 0
4. plugin dir 解決${TWILL_PLUGIN_DIR:-$CLAUDE_PROJECT_DIR/plugins/twl} で plugin path 取得env override 可
5. tmux new-windowtmux new-window -d -n <name> -c <cwd> -e TWL_PHASER_NAME=<name> -- "claude --plugin-dir <path>"-d=detached / -n=name / -c=start-directory / -e=env variable (tmux man verified)
6. session ready wait_wait_session_ready <window> 30 で 3 条件 AND 判定 (§3.3) を最大 30 秒timeout で _fail
7. slash command injecttmux send-keys -t <window> "/${skill} ${args}" Enter (Enter は C-m として送信)send-keys は spawn 直後 1 回のみ許容 (Inv I-7 例外)
8. spawn-completed mailmailbox_emit "$mailbox_from" "administrator" "spawn-completed" "{window, skill}"Inv T flock atomic write

env 注入 (step 5): TWL_PHASER_NAME=<window_name> を tmux -e 経由で session env に登録する。atomic SKILL.md Step 3 が mailbox path 解決にこの env を使用する (atomic-verification.html §3.1 参照)。

3.3 _wait_session_ready 仕様 (3 条件 AND 判定)

spawn 直後の claude session が ready かつ idle な状態を 3 条件 AND で判定する (false ready 防止)。判定間隔は 1 秒、default timeout 30 秒:

条件判定失敗時の動作
(a) pane idletmux list-windows -a -F '#{window_name}|#{pane_in_mode}' で pane_in_mode == 0 を assertidle_accumulated reset + 1 秒 sleep + retry
(b) inject queue 不在tmux capture-pane -t <window> -p -S -5 | tail -3 で "Press up to edit" 不在を assertqueue 残留検知 → Enter 送信して flush + retry
(c) idle 持続時間idle_accumulated >= IDLE_MIN_SEC (default 2 秒) を assert累積不足なら + 1 秒 sleep + retry

3 条件 AND 成立で return 0 (ready)、timeout 経過で return 1 (caller は _fail)。startup initialization 中の false ready 防止のため (c) idle 持続時間を必須化する。

verify points: EXP-010 (plugin 起動 + namespace 解決)、EXP-016 (tmux new-window -d)、EXP-017 (format 変数 pane state)。

§5. mailbox.sh helper 仕様 (~40 行)

API:
  • mailbox_emit <from> <to> <event> <detail_json> — atomic write 1 line
  • mailbox_read <mailbox-name> [--since <line-no>] — 未読 mail を stdout に出力 (read line-no を .read-offset file に保存)
  • mailbox_drain <mailbox-name> <callback-fn> — 未読 mail を 1 件ずつ callback に渡す
  • mailbox_archive <mailbox-name> — inbox.jsonl を archive/ に移動 (Wave 完了時)
関数動作仕様verify rule
mailbox_emitJSON Lines 1 行 ({from, to, ts, event, detail}) を jq -nc で生成し、$MAILBOX_ROOT/$to/inbox.jsonl に追記。flock -x -w 10 で exclusive lock 取得 + timeout 10 秒Inv T (atomic write)、fd 9 経由
mailbox_read$MAILBOX_ROOT/$mailbox/.read-offset から既読 line 数を取得、tail -n +$((from+1)) inbox.jsonl で未読を stdout に出力。読了後 wc -l < inbox で offset 更新。flock -s -w 10 で 共有 lockwrite との衝突防止 (exclusive)
mailbox_drainmailbox_read の各 line を引数 callback 関数に渡す (iteration helper)callback exit code を集計
mailbox_archive$MAILBOX_ROOT/$mailbox/inbox.jsonl を $MAILBOX_ROOT/archive/inbox-<mailbox>-<ts>.jsonl に mv。新 inbox は空作成Wave 完了時に呼ぶ

config: MAILBOX_ROOT=${TWILL_MAILBOX_ROOT:-.mailbox} で env override 可。実装規模は ~40 行 (set -euo pipefail + 4 関数 + lock file 管理)。

6.1 event 種別 (本 spec で固定):

命名規約: mail event 名の phase-* / step-* prefix は canonical event 名として継続、role prefix (phaser-* / workflow-* / atomic-*) とは独立な名前空間。twl audit --vocabulary の forbidden 検出は role prefix 用、event 名 prefix は exclusion_context として除外する。
  • spawn 系: spawn-completed, spawn-failed, spawn-skip
  • phase 系: phase-started, phase-progress, phase-completed, phase-failed (canonical event 名、role prefix とは独立)
  • step 系 (atomic→phaser): step-started, step-completed, step-precheck-failed, step-postverify-failed (mail event 名の step-* prefix は canonical event 名として継続、role prefix とは独立)
  • gate 系: phase-gate-denied
  • tool 系: tool-completed (tool-sandbox-runner のみ administrator に送る)
  • escalate 系: escalate-human-gate, budget-pause

§7. 3 階層全方向の使用例

7.1 atomic → phaser (step 完了 mail)

atomic SKILL.md Step 3 (report phase) が TWL_PHASER_NAME env 経由で recipient を解決し、mailbox_emit を呼ぶ:

7.2 phaser → administrator (phase 完了の集約 mail)

phaser SKILL.md が atomic/specialist mail を集約後、administrator mailbox に要約 mail を送る:

field値

to"administrator"
event"phase-completed"
detail{issue: 1660, status: "Implementing", atomic_count: 3, all_steps_pass: true}

7.3 administrator → phaser (spawn 経由、mailbox は使わない)

administrator が phaser を起動するのは spawn-tmux.sh 経由 (mailbox 不使用)。phaser 起動後の通信は atomic/specialist→phaser→administrator の上方向のみ (pyramid 集約、Inv I-4)。

§8. mail loss 防止 + archive 戦略

  • file 永続: disk 故障以外で loss しない (verified、JSON Lines + append-only)
  • backup: .mailbox/ は .gitignore 対象、git で持たない。重要 mail は archive で残す
  • archive: Wave 完了時に mailbox_archive で .mailbox/<name>/inbox.jsonl → .mailbox/archive/inbox-<name>-<ts>.jsonl
  • retention: 7 日 retention、cron で古い archive 削除
  • audit 統合: TWL_AUDIT=1 時に audit_snapshot(.mailbox, "mailbox-pre-archive") で .audit/<run-id>/ にもコピー

§10. subagent nesting 制約 (公式 verified)

公式制約 (verified): Claude Code subagent は 1 階層内側のみ spawn 可能、子 subagent の更なる spawn は禁止。
  • L2-3 (workflow / atomic / specialist) は内部で codex CLI 等の subagent を 1 階層内に持てる
  • L2-3 内 subagent が更に subagent spawn は 禁止
  • これが tmux で別 main session を spawn する根拠: 3 層 (administrator / phaser / L2-3) は全て別 main session、subagent 入れ子では実現不可
verified source: Claude Code 公式 docs (code.claude.com/docs/en/sub-agents.md L60/L348/L737)

§11. 不変条件 (本 file が enforce)

  • Inv T (file mailbox atomic write): 全 mailbox write は flock 取得した atomic write EXP-006
  • Inv V (per-specialist checkpoint): .mailbox/<session-name>/ は per-session、共通 path 禁止 EXP-013
  • Inv I-4 (administrator は phaser mail のみ受信): pyramid 集約、L2-3 (workflow/atomic/specialist) から administrator 直接 mail 禁止
  • Inv I-7 (tmux send-keys 使用禁止): ただし spawn-tmux.sh の spawn 直後 1 回のみ例外として inject 許可 (skill 起動 trigger)。それ以外の通信は mailbox 経由
