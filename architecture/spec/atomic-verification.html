§3. atomic SKILL.md inline 実装 standard

旧 step.sh framework の外部 bash 呼び出しを廃止し、atomic SKILL.md 本文に Step 0〜Step 3 として 4-phase を inline 実装。公式仕様 (disable-model-invocation: true + allowed-tools: [Bash, Read, Edit, Agent]) に準拠する。

§3.0 LLM 生成 vs hardcode 境界: atomic SKILL.md の各 Step は以下の分離原則に従う。
  • Step 0 (pre-check): SKILL.md に hardcode。固定 bash コマンド (test count snapshot / status verify / preconditions) で deterministic、LLM 介入なし
  • Step 1 (exec): LLM 自由生成。AC file を Read し、AC 内容に従って RED test or GREEN 実装を生成 (Edit/Write)。本 Step のみ non-deterministic を許容
  • Step 2 (post-verify): SKILL.md に hardcode。固定 bash コマンド (test count 増加 / RED 確認 / src diff 確認) で deterministic 検証、LLM 判断不可
  • Step 3 (report): SKILL.md に hardcode。固定 jq コマンドで mailbox に event 1 行追記、LLM 判断不可
判断根拠: deterministic な検証・送信は LLM の non-determinism を排除して experiment-verified しやすくする。Step 1 のみ LLM 自由 = 「LLM は判断のために使う、機械的にできることは機械に任せる」原則 (CLAUDE.md global) と整合。
§3.1 mailbox identifier 注入: atomic SKILL.md の Step 3 が emit する mailbox path の <phaser-name> 解決は $TWL_PHASER_NAME env 変数を経由する。
  • spawn-tmux.sh 責務: phaser tmux window 起動時に export TWL_PHASER_NAME=<phaser-name> を set (詳細: spawn-protocol.html §3)
  • atomic SKILL.md 参照: Step 3 の bash コマンドは .mailbox/$TWL_PHASER_NAME/inbox.jsonl を参照
  • 不在時の挙動: TWL_PHASER_NAME 未設定なら Step 0 で exit 1 (pre-check fail として早期 abort)
§3.2 Step 2 post-verify verdict policy: Step 2 の post-verify が判定する verdict は "PASS" 以外は全て exit 1 として phaser に fail 通知 (厳格 binary)。
  • specialist が Agent return value で "PASS" 以外 (FAIL / WARNING / unknown / 空文字 等) を返した場合は全て fail 扱い
  • bash 機械検証で grep / exit code が non-zero なら exit 1
  • "WARNING" を中間状態として通すと verify_checks の意味が曖昧化、phaser 側で 3 値判定の複雑さ増加 → 厳格 binary (PASS / FAIL) を採用
  • specialist の minor finding は findings_count: N として report に含めるが、verdict は PASS 維持 (specialist が独立判断、phaser は findings_count のみ判断材料に)

# atomic-test-scaffold

`Step 0` lifecycle phase: pre-check
- !`bats --list tests/ | wc -l > /tmp/pre_test_count`
- !`git diff --name-only HEAD | grep '^src/' | wc -l > /tmp/pre_src_diff`
- Verify `status` == Refined via !`gh project item-list ...`

`Step 1` lifecycle phase: exec
- Read $ARGUMENTS[0] (AC file for Issue)
- Generate RED test targeting the AC, Write to tests/unit/...
- (機械検証は Step 2 が担う、LLM はここで判断/作業のみ)

`Step 2` lifecycle phase: post-verify
- !`POST=$(bats --list tests/ | wc -l); PRE=$(cat /tmp/pre_test_count); [ "$POST" -gt "$PRE" ] || exit 1`
- !`bats tests/unit/... 2>&1 | grep -q "not ok"` # RED 確認 (fail が出ること)
- !`cat /tmp/pre_src_diff | grep -q '^0$'` # src/ 変更 0 確認

`Step 3` lifecycle phase: report
- !`jq -nc '{from:"atomic-test-scaffold",ts:"'"$(date -u +%FT%TZ)"'",event:"step-completed",detail:{red_count:N}}' >> .mailbox/$TWL_PHASER_NAME/inbox.jsonl` # §3.1 env 変数経由

disable-model-invocation: true + allowed-tools の rationale (verified skills docs):

  • disable-model-invocation: true により description が context budget に乗らない (EXP-035 で実測検証)
  • allowed-tools: [Bash, Read, Edit, Agent] — post-verify phase に ! prefix の Bash call が必須。Agent は specialist 呼び出し時のみ (EXP-036 で検証)
