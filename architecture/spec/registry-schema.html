§10. verify status defining criteria (4-state lattice)

「AI 自己申告 false positive」を構造的に潰す目的で、spec 内 <span class="vs ..."> 4 値 (inferred / deduced / verified / experiment-verified) の defining criteria を本節で SSoT 化する

10.1 4 status の defining criteria

StatusDefining criteria必要 evidence例
inferred 推測のみ、verify source 不明 / 未確認 なし (設計案のみ) 設計初期段階、spec 起草時の暫定記述
deduced spec / docs / Issue を読んで推論 (直接記述なし、間接 deduce) spec file 参照 / 推論経路の根拠 「公式 docs に X が書いてあるから Y も成立するはず」型の推論
verified 公式 docs / 仕様に直接明記、情報源 reachable verify_source URL (WebFetch / WebSearch で 200 OK + domain whitelist 内)、または内部 spec の §xxx 直接記述 公式 hook docs の matcher 仕様 / tmux(1) man の format 変数記述
experiment-verified 実機 empirical evidence (bats / smoke で動作確認 + log) result JSON 内に pass=true + log_hash + verify_checks (全 pass) bats で bash nameref 実機実行確認 / smoke で gh CLI 動作 + 出力 grep

10.2 status transition lattice (単方向、降格は audit 発覚時のみ)

stateDiagram-v2
  [*] --> inferred: 設計初期
  inferred --> deduced: spec / docs 読みで推論
  deduced --> verified: 公式 docs / 仕様で直接確認
  verified --> experiment_verified: 実機 empirical evidence (bats / smoke pass=true + log_hash + verify_checks)
  inferred: inferred (推測のみ)
  deduced: deduced (spec 推論)
  verified: verified (公式明記)
  experiment_verified: experiment-verified (実機 empirical)
  • 昇格: より強い evidence が得られた時のみ upgrade (例: 公式 docs で確認できたら deduced → verified、実機検証できたら verified → experiment-verified)
  • 降格: audit で「evidence が不足」と判明する場合のみ、user 確認下で実施 (例: verified と記載されているが URL が架空 → inferred に戻す)
  • 跳躍禁止 (default): deduced → experiment-verified のような中間 skip は default 禁止。必ず verified を経由する (公式明記なしで実機検証だけは矛盾)

10.2.1 multi-stage upgrade pattern

実装で発生し得る複数段昇格 case (例: EXP-019/020/021/022/023 の inferred → experiment-verified、delta=3) を踏まえ、以下の multi-stage upgrade pattern を user 判断下で例外的に許容する:

  1. 1 commit で 2-3 段上げる multi-stage upgrade: 各段階の evidence (verified の verify_source URL reachable + experiment-verified の smoke pass=true + log_hash + verify_checks 全 pass) が同 commit 内で完成する場合に限り、commit message + experiment-index.html status 注釈で各段階の evidence を明示することで、跳躍 commit を許容する
  2. audit 必須: experiments/audit-status-log.py (Phase G) が delta>=2 の transition を CRITICAL 記録。user が「multi-stage upgrade pattern として正当」と判断した場合のみ commit accept (この judgment は手動で audit log 上に annotation する)
  3. 例外的特殊 case: API_LIMITATION findings: smoke 実機実行で「期待する API が存在しない」ことを empirical 確認した場合 (EXP-022/023 の updateProjectV2View mutation 不在のように)、verify_source URL reachable + smoke pass=true (API_LIMITATION 内容を verify_checks に明示記録) が両立する。この場合 verified を経由せず直接 experiment-verified に upgrade して良い (verified の defining criteria 「公式 docs / 仕様に直接明記」が API 不在で不適用となる例外)
  4. multi-stage upgrade の禁止 case: inferred → experiment-verified で smoke 結果が verify_checks に server_state pass を含まない場合は、跳躍を絶対認めない (audit CRITICAL のまま、降格 revert 必須)

commit message annotation 例: "EXP-019: multi-stage upgrade (inferred → verified → experiment-verified, 1 commit): verify_source reachable + smoke pass=true + log_hash + verify_checks 全 pass で 2 段階同時 evidence"

10.3 status と必要 audit infrastructure

  • verified の audit: experiments/verify-source-check.py で manifest.json の verify_source URL を HTTP HEAD 200 OK + domain whitelist で検証。架空 URL を構造的に排除
  • experiment-verified の audit: experiments/audit-bats-quality.py で bats file の skip ratio / assertion count を計測。100% skip / dummy assertion を構造的に block
  • status transition log: experiments/audit-status-log.py (Phase G) で experiment-index.html の status edit を git diff として記録、 result JSON pass=true と乖離した手動 upgrade を検出

10.4 schema 静的 check / mock 単体は verified 止まり

bats unit test で JSON schema fixture 検証 / script ローカル動作確認 を行っても、実機 hook fire / Claude runtime 統合まで検証していなければ experiment-verified ではない。schema check は「公式 docs の claim を fixture で再表現できた」レベル (= verified の root evidence と同等) であり、empirical evidence (実機動作確認 + log) が揃って初めて experiment-verified に昇格する。
