twill plugin rebuild — crash failure mode

status: done · 3 階層 (administrator/phaser/L2-3) crash 検知 + recovery + SLA · 不変条件 G の強化

1. 目的

新 spec の 10 role architecture (administrator / phaser / workflow+atomic+specialist) の各層について、crash failure mode を仕様化する。既存 不変条件 G (クラッシュ検知保証) を強化し、各 crash の検知方法・recovery 手順・SLA を明示する。

背景: 既存 plugin の crash-detect.sh + pilot-fallback-monitor.sh (旧 script、現 phaser-level fallback 相当) は旧 worker crash (現 L2-3 specialist/atomic 層) のみカバー。旧 pilot crash (現 phaser) / administrator crash 時の recovery が不明確だった (failure-analysis.html の Bug #1674 関連)。本 spec で 3 階層全てを仕様化する。

4. crash 検知 SLA

層heartbeat 間隔timeout (検知閾値)検知者
L2-3 workflow+atomic+specialist3 min (各 atomic exec 終了時 touch)10 min (mail 不在 + heartbeat 古い)L1 phaser (mailbox poll cycle で確認)
L1 phaser5 min (各 cycle で touch)15 minL0 administrator (CronCreate 25 min cycle で確認)
L0 administrator25 min (CronCreate fire ごとに touch .supervisor/admin-heartbeat)10 min stale (§4.5 machine-scoped cron による短縮)machine-scoped Linux crontab (admin process と独立) + user 手動 fallback

SLA 設計の根拠

  • L2-3 atomic/specialist 3 min / 10 min: 単純作業 (test scaffold / impl) は通常 5-7 min、10 min 超過は hang と判定して妥当
  • L1 phaser 5 min / 15 min: atomic poll cycle (5 min) と整合、15 min は atomic 1 個分の hang 検知後の余裕
  • L0 administrator 25 min / 40 min: CronCreate fire 間隔 (25 min) に同期、1 回 skip を許容して 40 min
注意: SLA は初期値。実機運用で atomic exec の特性 (大規模 test execution で 10 min 超 等) に合わせて atomic ごとの timeout を個別調整する (atomic SKILL.md の timeout field 設計は rebuild-plan.html 参照)。

4.5 L0 administrator crash 自動検知 (Inv G 部分保証化)

問題: L0 administrator が crash すると CronCreate task も巻き添えで停止 (session-scoped)、L0 自己検知機構は原理的に成立しない (循環依存)。2 段防御を採用する。
1 段目 (session-scoped、admin 内)
admin polling cycle ごとに .supervisor/admin-heartbeat を touch。admin が alive な限り mtime 更新、admin dead で mtime stale
2 段目 (machine-scoped、admin 外)
Linux crontab (admin process と独立) で 5 min 毎に .supervisor/admin-heartbeat mtime check。10 min 以上 stale なら Discord DM via mcp__discord__reply (chat_id は reference 系 memory 参照) で user escalate。同 stale を複数回通知しないよう last-notified state file で idempotent 化する。
判断根拠: machine-scoped cron は admin process と独立で稼働、admin crash で fire 不能にならない。session-scoped CronCreate のみでは循環依存で検知不能 (admin dead → cron も dead)、machine-scoped で外部観測点を確保する。 残課題 (本質的に解消不能): 完全 crash 検知は不可能 (host down / network down / Discord MCP server down / user 通知経路 down のいずれかで検知 break)。Inv G は「部分保証 (host alive + Discord MCP alive な範囲で 10 min 内検知)」として認識する。

6. 周辺 failure mode

mail loss

  • 状況: L2-3 (atomic/specialist) が mail を書いて exit した直後に system crash (mail は file に persistent)
  • 対応: file mailbox は disk 永続なので mail loss は起こらない (不変条件 T: flock atomic write 必須)。phaser crash 時の再起動でも mail は読める。
  • 例外: disk 故障の場合は別問題、本 spec の scope 外 (ADR で扱う)。

status race (phaser 更新 vs administrator poll)

  • 状況: phaser が GitHub Project Board status を更新 + mail を書く間に administrator poll が走る
  • 対応: administrator は mail と status の両方を確認、不整合 (mail なし + status 変化) の場合は 1 cycle 遅延 + 再 poll
  • 不変条件 reference: admin-cycle.html の不変条件群を参照。本 spec では「mail と status の AND check」原則を採用する

CronCreate skip / delay

  • 状況: admin の CronCreate が claude code internal で skip / delay
  • 対応: CronCreate `durable=true` は BROKEN (#40228) のため session 再起動で fire 不能 (admin-cycle.html §1 参照)。tmux session 再起動時は administrator が --continue で復帰し CronCreate を再登録する。skip 検知は heartbeat file mtime > 10 min で machine-scoped cron (§4.5) が user notify する

API rate limit / external service 障害

  • 状況: GitHub API / claude code API rate limit、MCP server 障害
  • 対応: 各 helper (mailbox.sh / atomic SKILL.md inline / phaser-gate.sh) は API 失敗時 exponential backoff retry (最大 3 回)、最終失敗で mail に明記して上位 layer に escalate

7. recovery 経路の不変条件 (新 spec で確立)

  • 不変条件 G 強化: 全 3 階層に heartbeat file + 上位 layer 確認 mechanism MUST
  • 新規 invariant X (deploy/verify セット): heartbeat-watcher.sh は 起動 hook + 起動確認テスト を同 PR で merge MUST
  • recovery 試行回数: 各層 1 回まで (不変条件 E と整合)、2 回目失敗で上位 layer に escalate
  • state 永続性: file mailbox + GitHub Project Board status の 2 SSoT で L2-3 (workflow/atomic/specialist) 状態は再起動後も復元可能
  • 巻き添え死防止: 各層は別 main session (tmux new-window) で隔離、L0 administrator crash でも L1/L2-3 は生存 (tmux server 全死の例外を除く)
