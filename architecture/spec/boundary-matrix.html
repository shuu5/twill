boundary-matrix — 10 role 責務 boundary (第 5 弾 dig 反映)

draft-v3 (2026-05-13、第 5 弾 dig 反映) · 第 4 弾 dig (tool-* 3 件) + 第 5 弾 dig (pilot→phaser、worker→workflow/atomic/specialist 分割、step.sh→atomic SKILL.md inline)
本 file は registry-schema.html §3 (10 role × concern matrix) の参照下にある。10 role の責務 boundary を行列形式で明示し、各 role が何を行う / 行わない / 委譲する かを clean table で定義。

目的

10 role (administrator / phaser / workflow / atomic / specialist + tool / reference / script / hook / monitor) の責務 boundary を行列形式で明示する。簡略化のため core 5 entity (administrator / phaser / workflow+atomic+specialist (L2-3) / tool / status-transition) を中心に列挙。

Phase I 2026-05-15 追加: hooks 全体規定 (Inv H-1〜3 hook independence、各 hook は他 hook 中間 state に依存しない、共有 state は file 経由のみ) は hooks-mcp-policy.html §3 で SSoT 化、本 file の各 entity section gate hook 行と整合する。Monitor primitive 規定 (actor scope + signal primary + composite + lifecycle) は monitor-policy.html で SSoT 化、本 file 各 entity の Monitor primitive 行と整合する。

5 entity boundary matrix (administrator / phaser / workflow+atomic+specialist / tool-* (3 件) / status-transition)

操作 administrator (L0) phaser (phaser-*、L1) workflow + atomic + specialist (L2-3) tool-* (3 件、独立) status-transition 概念
spawn 元 user 直接起動 administrator (admin-cycle.sh) phaser から Skill() (workflow/atomic) または Agent() (specialist) user 明示 (tool-architect / tool-sandbox-runner は admin も可) phaser SKILL.md として動作
spawn 先 phaser + tool-architect / tool-sandbox-runner (限定、ADR-040 gate) workflow / atomic (Skill()) + specialist (Agent()) specialist は spawn 禁止 (公式制約 verified)、workflow/atomic は Skill() chain 続行 + Agent(specialist-*) 可 (spawn 禁止) —
status 遷移 read only (Inv I-5) 遷移実行 (gh project item-edit)、1 status-transition を 1 phaser が担当 (更新禁止、phaser 経由) 無関係 (MUST NOT 触れる) 1 status-transition を 1 phaser が担当 (forbidden: state)
mailbox phaser mail のみ受信 (Inv I-4 pyramid) atomic/specialist mail 集約 + administrator に要約 mail atomic は phaser mailbox に書き込み、specialist は subagent context で結果 return tool-sandbox-runner のみ administrator mail (他 2 tool は user 直接) —
worktree (操作禁止) atomic 用 worktree 作成 + 完了時削除 (Inv B) atomic は自身の worktree 内で実作業、specialist は worktree 不要 (subagent context) tool-architect が PR 作成時に独自 worktree —
verification framework (利用しない) workflow を Skill() で呼ぶ、各 atomic の 4-phase post-verify を確認 atomic SKILL.md 本文に 4-phase lifecycle (pre-check/exec/post-verify/report) を inline 実装 (Inv U "Atomic skill verification") (利用しない、status と無関係) —
gate hook (対象外) PreToolUse gate 適用 (Inv W) — phaser invocation 時に前提 status verify (対象外、Skill()/Agent() 公式機構の内側) (対象外、status と無関係) —
TWL_AUDIT 出力 .audit/<run-id>/admin-cycle.jsonl .audit/<run-id>/phaser/<name>/ .audit/<run-id>/atomic/<name>/ + .audit/<run-id>/specialist/<name>/ .audit/<run-id>/tools/<name>/ —

本 table は admin-cycle.html / spawn-protocol.html / atomic-verification.html / gate-hook.html / tool-architecture.html と整合する verified content。

各 entity の責務詳細

administrator (L0) の責務

spawn 元
user 手動起動 (+ --continue/--resume で session 継続)。Desktop scheduled tasks で machine boot 自動起動も候補
spawn 先
phaser (phaser-* 4 種) + tool-* 3 件 (限定)。spawn-tmux.sh 経由、Inv I-1 (idempotent) + I-2 (parallelism 上限) を守る
status 遷移
read only (Inv I-5)。gh project item-list で 5 min cache poll。write は絶対禁止 (user 明示介入の例外のみ)
mailbox
.mailbox/administrator/inbox.jsonl のみ受信 (Inv I-4 pyramid)。atomic/specialist 直接 mail 禁止、phaser 集約後に administrator。flock 共有 lock で read
worktree
操作禁止 (read のみ)。git worktree コマンド実行禁止、phaser 経由
verification framework
利用しない (administrator は polling cycle 6-step、atomic SKILL.md inline lifecycle は atomic 専用)
gate hook
対象外 (administrator は phaser-* invocation でないため PreToolUse gate 発火しない)
audit 出力
.audit/<run-id>/admin-cycle.jsonl に全 cycle の read/judge/write 記録
受信 mail event
phase-completed / phase-failed / phase-gate-denied / spawn-completed / tool-completed (tool-sandbox-runner のみ)
送信 mail event
spawn-request / escalate-human-gate / budget-pause
escalate
3 連続 failure → ★HUMAN GATE (AskUserQuestion or Discord mail)。budget exhaustion → CronDelete + one-shot wakeup
crash recovery
heartbeat self-touch (Inv G)、session 死亡時は user 手動 --continue。CronCreate 復元 (7 日 expiry 注意)。Phase I 2026-05-15 追加: .supervisor/admin-heartbeat file を admin polling cycle 毎に touch、stale 検知時 (10 min 以上更新なし) は CronCreate 別 task が Discord DM (mcp__discord__reply) で user escalate (Inv G 部分保証化、詳細 crash-failure-mode.html §3)
Monitor primitive
.mailbox/administrator/inbox.jsonl tail + budget status line + tmux state を Monitor で集約 (monitor-policy.html §2)。plugins/twl/monitors.json daemon を所有 (Step 0 で health-check、crash 時 record-detection-gap.sh で user escalate)

phaser (phaser-*、L1) の責務

spawn 元
administrator (admin-cycle.sh step 4)。spawn-tmux.sh wrapper 経由
spawn 先
workflow-* (Skill()) + 必要時 specialist-* (Agent())。atomic は workflow から呼ばれる
status 遷移
遷移実行専任。gh project item-edit --single-select-option-id を 1 回のみ実行。担当 status-transition の 1 遷移のみ責任 (phaser-impl なら Refined → Implementing)
scope
1 phaser = 1 Issue・1 status-transition 専任 (Phase I dig Q3 確定 2026-05-15)。複数 Issue の sequential 処理や複数 status-transition の連続処理は MUST NOT。mailbox lifecycle = phaser invocation lifecycle (担当完了で .mailbox/<phaser-name>/ を .mailbox/archive/ に archive)
mailbox
受信: atomic/specialist から .mailbox/<phaser-name>/inbox.jsonl。送信: administrator mailbox に要約 mail のみ (pyramid 集約)。送受信両方を行う唯一の layer。<phaser-name> 解決は TWL_PHASER_NAME env 経由 (spawn-protocol.html §3 + atomic-verification.html §3.1)
worktree
lifecycle 専任 (Inv B)。git worktree add (phaser 開始時) + git worktree remove (phaser 完了時)。atomic/specialist は worktree を自ら作成・削除しない
verification framework
workflow を Skill() で呼び、各 atomic の 4-phase post-verify 結果を mailbox から確認。phaser 自身は atomic SKILL.md 本文を実行しない
gate hook
PreToolUse gate 対象 (Inv W)。gate hook handler (旧 phase-gate.sh) (tier 1) + twl_phase_gate_check (tier 2) が phaser invocation 時に発火、前提 status 不一致なら deny + administrator notify
audit 出力
.audit/<run-id>/phaser/<name>/ 配下に phaser lifecycle 全 event 記録
送信 mail event
phase-started / phase-progress / phase-completed / phase-failed / merge-gate-failed
review framework
atomic 出力検証 (post-verify 結果確認、src diff、test count 増加確認)。Inv K (phaser 実装禁止、第 5 弾 dig で phaser に rename) で src diff = phaser のみは FAIL
merge gate
phaser-pr のみが PR merge を実行 (Inv C 保全)。他 phaser は merge 禁止
内部 spawn
Agent(specialist-*) は可 (公式 verified、subagent context)、subagent からの更なる spawn は禁止
Monitor primitive
atomic spawn 後の .mailbox/<own-phaser-name>/inbox.jsonl tail で進捗監視可 (monitor-policy.html §2)。制約: administrator と同 channel を tail しない (token 二重消費回避)、channel 分離 (admin = phaser 集約 mail / phaser = atomic mail)、phaser invocation 終了時に Monitor task stop

workflow + atomic + specialist (L2-3) の責務

spawn 元
workflow / atomic: phaser または上位 workflow から Skill() 公式機構。specialist: phaser または atomic から Agent() 公式機構
spawn 先
specialist は spawn 禁止 (subagent spawn 不可 verified)。workflow/atomic は Skill() chain 続行可、atomic 内で Agent(specialist-*) 可
status 遷移
更新禁止。phaser 経由のみ。gh project item-edit 実行は MUST NOT。atomic/specialist は phaser に mail で報告し、phaser が status を遷移
mailbox
atomic は phaser mailbox (.mailbox/<phaser-name>/inbox.jsonl) にのみ書き込み (4-phase report で step-* event を emit)。specialist は subagent context で結果 return、mailbox 書き込みなし
worktree
atomic は自身の worktree 内での実作業のみ (cwd-guard Inv B)、phaser が作成した worktree 内で git checkout/edit/test 実行。他 Issue の worktree に触れることは MUST NOT (Inv V)。specialist は worktree 不要 (Read only)
verification framework
atomic SKILL.md 本文に 4-phase lifecycle (pre-check → exec → post-verify → report) を inline 実装 MUST (Inv U "Atomic skill verification")。self-report-only 禁止。workflow は atomic を順次 Skill() で呼ぶ launcher (verify 詳細 inline なし)。specialist は独立判断、自己 verify は subagent prompt 内で完結
gate hook
対象外。Skill()/Agent() 公式機構の内側で PreToolUse gate 発火しない。gate は spawn する phaser の invocation 時に発火済み
audit 出力
atomic: .audit/<run-id>/atomic/<name>/ 配下に 4-phase 全 event 記録。specialist: .audit/<run-id>/specialist/<name>/ 配下に Agent return 記録
送信 mail event
atomic: step-started / step-completed / step-precheck-failed / step-postverify-failed。specialist: mail 送信なし (Agent return value で完結)
内部 spawn
atomic 内 Agent(specialist-*) 可、specialist は spawn 禁止 (公式 verified)
checkpoint
.mailbox/<phaser-name>/ または per-phaser パスが必須 (Inv V)。共通 path への write は --shared フラグ明示 + audit log 必須
Monitor primitive
使用不可 (monitor-policy.html §2)。run-to-completion 型のため Monitor 不要、Skill() / Agent() の return value または mail emit (atomic のみ) で十分

tool-* (3 件、独立) の責務 (第 4 弾 dig)

詳細は tool-architecture.html §2-§5 + dig-report-tools-2026-05-13.md §1-§3 参照。要約:

3 tool 構成
tool-architect (HTML spec 維持 + 軽量 PR cycle) / tool-project (GitHub Project + template + verify の 3 軽 scope) / tool-sandbox-runner (sandbox × feature matrix + Idea Issue 起票)。第 4 弾 dig で tool-utility 廃止 (commands は他 3 tool + administrator inline に再分配)
spawn 元
user 明示 (slash command /twl:tool-architect <args>) または administrator (tool-architect / tool-sandbox-runner のみ、ADR-040 gate 通過後)
spawn 先
spawn 禁止 (内部 Agent(specialist-*) は可、1 階層)
status 遷移
無関係 (MUST NOT 触れる)
mailbox
tool-sandbox-runner のみ administrator に tool-completed mail。他 2 tool は user に直接結果提示 (mailbox 不使用)
worktree
tool-architect は spec edit PR の独自 worktree、tool-sandbox-runner は test-target/main orphan branch の専用 worktree (verified pattern 継承)
verification framework
利用しない (status と無関係)
gate hook
対象外 (status と無関係、ADR-040 gate が SSoT)。例外: tool-architect は別系統で spec edit boundary hook (pre-tool-use-spec-write-boundary.sh) を保持し、tool-architect / user 以外の Edit/Write を deny
audit 出力
.audit/<run-id>/tools/{architect,project,sandbox-runner}/ 配下
自律 verify
tool-architect は 2 層 verify (SKILL.md MUST + PostToolUse hook verify-coverage.sh warn) で 4-state status の漏れを catch、軽量 PR cycle で changelog 漏れを specialist-spec-review agent が検出 (3 回 fix loop 上限)
