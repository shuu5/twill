§1. 公式 hook schema (verified)

公式 schema の重要 point (verified、誤用しやすい箇所):
  • "matcher" (単数) を使う。"matchers" (複数形) は採用しない
  • {{tool.skill_name}} / {{matcher.expected_status}} の template 変数展開は 非サポート。すべての context は stdin JSON で渡される
  • exit 1 は non-blocking error (= allow)、deny は exit 2 または exit 0 + JSON 出力で表現する
  • permission decision の enum は permissionDecision: "allow"|"deny"|"ask"|"defer"。decision: "approve" / "block" 形式は採用しない

1.1 正しい hook config schema (公式 verified)

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Skill",
        "hooks": [
          {
            "type": "command",
            "if": "Skill(phaser-impl*)",
            "command": "${CLAUDE_PROJECT_DIR}/plugins/twl/scripts/phase-gate.sh",
            "args": [],
            "timeout": 5
          },
          {
            "type": "mcp_tool",
            "if": "Skill(phaser-impl*)",
            "server": "twl",
            "tool": "twl_phase_gate_check",
            "input": { "expected_status": "Refined" }
          }
        ]
      }
    ]
  }
}
公式引用: 「The key is matcher (singular), not matchers. It appears at the matcher group level. The matcher narrows by tool name. The inner hooks array contains one or more handler objects.」
source: code.claude.com/docs/en/hooks

1.3 if field の syntax (verified、permission rule)

if field は permission rule syntax を使う、matcher の追加絞り込み。tool 系 event でのみ有効 (PreToolUse / PostToolUse / PostToolUseFailure / PermissionRequest / PermissionDenied)。

  • "Bash(git *)" — Bash の git subcommand
  • "Edit(*.ts)" — TypeScript file の Edit
  • "Skill(phaser-impl-*)" — phaser-impl で始まる skill 全件

§3. deny/allow signal (公式 verified)

Exit Code意味tool 実行
0success。stdout を JSON として parse。JSON 無し or permissionDecision: "allow" → 許可。"deny" / "ask" / "defer" → それぞれの挙動JSON 次第
2blocking error。stderr が Claude に表示、tool call は denydenied
その他 (1, 3...)non-blocking error。tool call は allow されるが error 記録allowed

3.1 JSON 出力 schema (exit 0 時、公式 verified)

{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow|deny|ask|defer",
    "permissionDecisionReason": "Human-readable reason",
    "additionalContext": "Optional context for Claude",
    "updatedInput": { "command": "modified command" }
  }
}

3.2 deny の最 restrictive 勝ち (公式 verified)

同 event の全 hook は parallel 実行、permissionDecision は deny > ask > allow の最 restrictive が勝つ。

重要: PreToolUse は permission mode より 先に発火、bypassPermissions や --dangerously-skip-permissions でも deny は効く。これが Inv W の構造的保証。

§4. tier 1 gate hook handler 仕様 (command hook、~40 行 規模)

tier 1 は粗フィルタ + fast path として動作する command hook handler。詳細判定は tier 2 (mcp_tool) に委譲し、tier 1 では 50 ms 以内 の早期 return を実現する。

項目仕様
inputstdin JSON (§2 で定義: tool_name / tool_input.skill / session_id 等)
抽出 field.tool_name、.tool_input.skill、.session_id (jq -r で安全に取得、未定義は空文字列 fallback)
粗フィルタtool_name != "Skill" または skill が phaser-* 形式でない場合は exit 0 (allow、JSON なし) で早期 return
bypassenv TWL_PHASE_GATE_BYPASS が非空の場合のみ permissionDecision: "allow" + reason 付き JSON で明示 allow (debug 用 escape hatch、§8 bypass policy 参照)
正常 pathtier 1 自身は判定せず exit 0 (pass)。tier 2 (MCP shadow) が deny を返した場合、deny > allow の最 restrictive 勝ち (§3.2) で全体 deny になる
outputexit 0 (JSON 省略時 allow) または permissionDecision field 付き JSON (§3.1 schema)
fail-safeset -euo pipefail、jq 失敗時も exit 0 (gate を fail open にして tool 実行を妨げない、tier 2 で確実 deny される設計)

§5. tier 2 twl_phase_gate_check 仕様 (MCP shadow、stateful 判定)

5.1 handler 仕様 (純 Python、in-process テスト可能)

§7. deny 経路 (administrator notify)

tier 2 が deny を返した場合、tier 1 gate hook handler が並列で administrator mailbox に notify mail を送る。mail schema は monitor-policy.html §3.2 の canonical schema に従う:

field値
from"phase-gate" (gate hook handler の固定 sender id)
to"administrator"
tsISO8601 timestamp (date -Iseconds)
event"phase-gate-denied"
detail{ skill: <skill_name>, issue: <issue_num>, expected: <expected_status>, actual: <current_status> }

書き込み path は .mailbox/administrator/inbox.jsonl、Inv T (flock atomic write) を遵守する (flock -x 経由)。

§8. bypass policy

  • 許可: TWL_PHASE_GATE_BYPASS=1 env (user 手動指定、debug 用)
  • 許可: 該当 Issue の Project Board status を user が手動で変更 (root cause 解消)
  • 禁止: code 内の SKIP_*_REASON や SKIP_PILOT_GATE 等の env var による bypass — これが Bug #1660/1662/1663 の根本問題

§9. retry path (前 status-transition 完了待ち or user 介入)

  1. deny 受信 → administrator が mail を読み、状況を user に報告
  2. user 判断:
    • 前 status-transition が未到達 → 該当 phaser-* を先に実行
    • status 遷移漏れ → user 手動で Project Board status 更新
    • 緊急 → TWL_PHASE_GATE_BYPASS=1 で bypass
  3. administrator polling cycle で status 変更を検知 → 再度 phaser-* invocation

§10. Bug #1660/1662/1663/1684 の構造的不能化

Bug既存 root cause新 spec での不能化
#1660 SKIP_*_REASON sanitize bash env で bypass、子プロセスへ自動継承、sanitize なし bypass env 全廃。PreToolUse hook + Project Board status SSoT が唯一の gate、env による緊急 bypass のみ TWL_PHASE_GATE_BYPASS で明示
#1662 :- syntax bash の ${VAR:-default} と ${VAR-default} の意味差で env override 無効化 env override 不要。phase-gate.sh は env を読まず、tier 2 が Project Board 直接 query
#1663 env override 競合 controller_count=0 短絡 path で env reassign、override 上書き env reassign の概念廃止。Python 関数引数として override (expected_status / skill_name) を渡す
#1684 IS_AUTOPILOT cwd-guard 「autopilot mode」と「正しい worktree」を 1 env で表現したガード漏れ 独立 invariant の別 enforce。tier 1 で skill 名フィルタ、tier 2 で status query、worktree 判定は別 hook (worktree-boundary.sh)

§11. verify 方法 (bats test)

11.1 phase-gate-hook.bats test scenarios

Inv W enforcement を verify する bats test の必要 scenario。stdin JSON を pipe で gate hook handler に流し、出力 JSON の permissionDecision を assert する:

scenariosetupexpected
phaser-impl on Refined Issue is allowedIssue Status = Refinedexit 0 + JSON なし (allow) or permissionDecision: "allow"

mock 戦略: _get_project_status 内の gh project item-list subprocess を bats helper でモックし、test fixture の status を returnable にする (mock_project_status 1660 "Refined" 形式の helper)。

11.2 verify points

rebuild-plan.html verify points に gate hook 関連として以下を追加:

  • tier 1 + tier 2 hook が parallel 実行され、deny > allow の最 restrictive 勝ち (§3.2) が成立する (additionalContext に両 tier の reason が含まれる)
  • bypassPermissions mode でも tier 2 の deny が効く (TWL_PHASE_GATE_BYPASS 未設定で実証)
  • status mismatch 時に administrator mailbox (§7) に phase-gate-denied mail が書き込まれる
  • gh project item-list キャッシュが 5 min TTL で invalidate される
