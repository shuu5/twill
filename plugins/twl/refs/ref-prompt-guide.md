# プロンプト執筆ガイド

**ref-practices** が「原則・パターン・設計ルール」を定義する上位ドキュメントであるのに対し、本ドキュメントは「個々のプロンプトファイルを書く・直す際の具体的ガイド」として ref-practices を補完する。

---

## 1. 型別コンテンツルール

### controller

| カテゴリ | 内容 |
|---------|------|
| **書くべきもの** | ワークフローのステップ順序・条件分岐・フロー制御ロジック、各ステップでの「何を呼び出すか」の指示 |
| **書いてはいけないもの** | データ加工ロジック、バリデーション実装、フォーマット定義、アーキテクチャ概要ドキュメント |
| **外部化すべきもの** | composite/atomic の詳細指示（Read で参照）、仕様・チェックリスト（reference へ）、ドメインルール（CLAUDE.md or reference へ） |

**サイズ目安**: 本文 ~80行推奨。120行超 → Warning、200行超 → Critical（即分割）。

---

### workflow

| カテゴリ | 内容 |
|---------|------|
| **書くべきもの** | フェーズ遷移ロジック、中間結果の引き継ぎ方法、分岐条件（IS_QUICK など） |
| **書いてはいけないもの** | specialist の詳細実行指示（composite に委譲）、設計ドキュメント |
| **外部化すべきもの** | 複雑な分岐判断（atomic コマンドに委譲）、再利用可能なチェック仕様（reference へ） |

---

### composite

| カテゴリ | 内容 |
|---------|------|
| **書くべきもの** | specialist 構成の定義（技術スタック判定テーブルなど）、Task spawn テンプレート（prompt・model・maxTurns）、結果統合ロジック |
| **書いてはいけないもの** | ワークフロー全体の制御ロジック（controller に帰属）、specialist の実装詳細 |
| **外部化すべきもの** | 判定基準（reference へ）、spawn されるプロンプトの詳細（specialist ファイルへ） |

---

### atomic

| カテゴリ | 内容 |
|---------|------|
| **書くべきもの** | 単一タスクの完全な実行手順（Step N 形式）、必須チェックリスト、終了条件 |
| **書いてはいけないもの** | specialist spawn 指示（composite へ昇格）、複数の独立した処理の統合 |
| **外部化すべきもの** | 繰り返し参照される判定基準（reference へ）、共通フォーマット定義（reference へ） |

---

### specialist

| カテゴリ | 内容 |
|---------|------|
| **書くべきもの** | タスク目的（1文）、使用ツール一覧、成功条件・出力フォーマット、エラー時の行動指針 |
| **書いてはいけないもの** | 他の specialist への spawn 指示（Task tool 禁止）、ワークフロー管理ロジック |
| **外部化すべきもの** | チェックポイント定義（reference へ、checkpoint_ref で参照）、再利用判定基準（reference へ） |

**必須フィールド**: `tools:`, `maxTurns:`, `effort:` を frontmatter に明記。

---

### reference

| カテゴリ | 内容 |
|---------|------|
| **書くべきもの** | 複数コンポーネントから参照される仕様・チェックリスト・判定基準・チェックポイント定義 |
| **書いてはいけないもの** | 特定コンポーネント専用の実行ロジック（atomicへ）、一度しか参照されない情報 |
| **外部化すべきもの** | なし（reference 自体が外部化の終端） |

**注意**: token_target なし（意図的除外）。サイズ制限なし。

## 2. トークン目標と構造的リファクタリング戦略

> **権威ソース**: token_target の値は `plugins/twl/types.yaml` の `token_target` フィールドを参照すること（Issue #103 完了後に有効化）。現時点の参照値は `ref-practices.md` の「トークン閾値」テーブルを使用。

### 現在の閾値（ref-practices.md より）

| 型 | Warning | Critical |
|----|---------|----------|
| controller | 1,500 tok | 2,500 tok |
| workflow | 1,200 tok | 2,000 tok |
| atomic | 1,500 tok | 2,500 tok |
| composite | 1,500 tok | 2,500 tok |
| specialist | 1,800 tok | 2,500 tok |

### 段階的対処フロー

```
超過検出
  │
  ▼
[Step 1] テキスト圧縮
  - 重複説明を削除、箇条書き化、例示を1つに削減
  - 目標: Warning 未満に収める
  - 成功 → 完了
  │
  ▼（まだ Warning 超え）
[Step 2] 外部化
  - 繰り返し参照される判定基準・チェックリストを reference に切り出す
  - 本文では「{reference名} を参照」と記載
  - 成功 → 完了
  │
  ▼（まだ Warning 超え または Critical 超え）
[Step 3] 分割
  - atomic: 独立したタスクが2つあれば2ファイルに分割（「分割判定基準」参照）
  - composite: 独立したフェーズが2つあれば phase-A/phase-B に分割
  - specialist: 独立したチェック観点があれば specialist を追加して並列化
  - 成功 → 完了
  │
  ▼（Critical 超え、分割しても解決しない）
[Step 4] 型変更
  - atomic → composite（specialist spawn 追加）
  - workflow → controller（セッション管理が必要になった場合）
  - 型変更後は deps.yaml の type・can_spawn・spawnable_by を更新
```

### 判定木

```
トークン超過？
├─ Warning 未満 → 対処不要
├─ Warning 以上 Critical 未満
│   ├─ テキスト圧縮で解決できる？ → Step 1
│   └─ 解決できない → Step 2（外部化）
└─ Critical 以上（即座に対処必須）
    ├─ 外部化で解決できる？ → Step 2
    ├─ 独立タスクが複数？ → Step 3（分割）
    └─ 型の責務が広すぎる？ → Step 4（型変更）
```

## 3. 分割判定基準

### atomic を2つに分割するとき

以下の条件を **すべて** 満たす場合に分割する:

| 条件 | 判定方法 |
|------|---------|
| 2つのタスクが **独立している**（順序不問、一方が他方の結果に依存しない） | タスクを逆順に実行しても問題ないか？ |
| 各タスクが **別々の文脈で再利用される可能性がある** | 他の workflow/controller から片方だけ呼ばれるユースケースがあるか？ |
| 分割後の各ファイルが **単独で意味をなす** | 読み手がもう一方を読まなくても理解できるか？ |

**分割しない基準**:
- 2つのタスクが強く依存している（Step 1 の結果を Step 2 で使う）
- 分割後のファイルが 5行程度になる（軽微すぎる）

---

### atomic → composite に昇格するとき

以下のいずれかを満たす場合:

| トリガー | 例 |
|---------|-----|
| 独立した観点で **並列チェックが有効** になった | セキュリティ・コード品質・型安全性を同時にレビューしたい |
| 1つの処理を **複数エージェントに分担** させたい | ファイルA担当 / ファイルB担当 |
| 処理が **大規模化** し、1セッションで完結させるとコンテキスト不足になる | 50ファイル以上の一括チェック |

**昇格後の作業**:
1. composite ファイルに specialist 構成・Task spawn テンプレートを記載
2. specialist ファイルを新規作成
3. deps.yaml で type を `composite` に変更、can_spawn・calls を更新

---

### workflow を分割するとき

以下の条件を満たす場合:

| 条件 | 判定 |
|------|------|
| ユーザーがワークフローの **途中から独立して実行したい** フェーズがある | 例: setup と test-ready を別々に呼べる |
| フェーズごとに **異なるコンテキスト量** が必要 | フェーズ1は軽量、フェーズ2は重い |
| 同じフェーズが **複数の controller から再利用** される | DRY 原則 |

**分割しない基準**:
- フェーズが3ステップ以下で全体が軽量
- フェーズ間の依存が強く、独立実行に意味がない

## 4. 5原則プロンプト単位チェックリスト

ref-practices.md の5原則を、個々のプロンプトファイルを**書く・直す**際に適用する具体的チェック項目。

### 完結 (Self-Contained)

- [ ] プロンプトの **目的が冒頭1文** で明記されている
- [ ] 成功条件（何ができれば完了か）が**具体的に定義**されている
- [ ] 前提条件（このプロンプトを使う前に必要なもの）が明記されている
- [ ] 使用するツールが `allowed-tools` / `tools` に**全て列挙**されている（過不足なし）
- [ ] 出力フォーマット（ファイルパス・形式・マーカー等）が定義されている
- [ ] エラー・ブロック時の行動指針が記載されている

---

### 明示 (Explicit)

- [ ] frontmatter に型・ツール・（specialist は model も）が宣言されている
- [ ] 禁止事項（MUST NOT / 禁止）が明記されている
- [ ] 曖昧な表現（「適切に処理」「必要に応じて」）が使われていない
- [ ] ステップが番号付きで順序明確に記載されている（atomic/composite）
- [ ] 参照する外部リソース（ファイルパス・コマンド）が具体的に明記されている

---

### 外部化 (Externalize)

- [ ] フェーズ間で引き継ぐデータの**保存先**が明記されている
- [ ] 繰り返し参照する判定基準は reference に切り出されている（インライン重複なし）
- [ ] 大量データ取得・Web フェッチは specialist / atomic に委譲されている
- [ ] controller 本文にデータ加工ロジックが**混入していない**

---

### 並列安全 (Parallel-Safe)

（composite・specialist が並列実行される場合のみ適用）

- [ ] 各 specialist の作業対象ファイルが**重複していない**
- [ ] 共有書き込みファイルがある場合、**書き込み責任者が1つ**に限定されている
- [ ] 結果統合ロジックが composite に**一元化**されている
- [ ] parallel 実行と sequential 実行の区別が明記されている

---

### コスト意識 (Cost-Aware)

- [ ] specialist frontmatter に `effort:` が設定されている（low / medium / high）
- [ ] specialist frontmatter に `maxTurns:` が設定されている
- [ ] タスク難度に対して**モデルが過剰**でないか確認した（構造チェックに opus 不要）
- [ ] コンテキストを大量消費する処理（Web フェッチ・大量ファイル読み込み）が Subagent Delegation されているか検討した
- [ ] トークン超過リスクがある場合、「2. トークン目標と構造的リファクタリング戦略」を参照した
