## Context

loom-plugin-dev は Phase 1+2 で全17 Issue を実装・マージ済み。その後コマンド形式リファクタ（COMMAND.md → \<name\>.md, cbac4ad）を実施したが、テストスイートの実行結果が未確認。テストは `tests/run-tests.sh` で一括実行され、37 bats テスト + 37 scenario テストの計74件が存在する。

## Goals / Non-Goals

**Goals:**

- ベースライン記録: 現在の PASS/FAIL 数を Issue #43 のコメントに記録
- 失敗テストの原因分析と修正（10件以下の場合）
- hooks（PostToolUseFailure）の動作確認
- `chain generate --check` の PASS 確認

**Non-Goals:**

- 新規テストの追加
- E2E レベルの動作検証（#44 のスコープ）
- テストフレームワーク自体の変更

## Decisions

1. **ベースライン先行記録**: 修正前に現状の PASS/FAIL 数を記録し、修正効果を定量的に把握する
2. **スコープ制限ルール**: 失敗テストが10件超の場合はベースライン記録+分類のみとし、修正は別 Issue に分割する
3. **修正アプローチ**: テストコード側の修正を優先し、プロダクションコードの変更は最小限にする。リファクタ（COMMAND.md → \<name\>.md）への追従が主な修正内容と想定
4. **hooks 検証**: PostToolUseFailure hooks が意図通りにエラーを検出・報告するか、テスト実行中に確認する

## Risks / Trade-offs

- **リスク**: 失敗テストが10件超の場合、本 Issue では修正完了せず別 Issue への分割が必要
- **リスク**: テストが参照するファイルパスがリファクタで変更されている可能性。grep でパス参照を確認してから修正
- **トレードオフ**: テストコード修正 vs プロダクションコード修正。テスト側で吸収できるものはテスト側で対応し、プロダクションコードのバグが見つかった場合のみプロダクション側を修正
