# E2E テスト計画作成

コードベースの静的分析により、テストすべきシナリオを特定する。

## 使用方法

```
/dev:e2e-plan <url|feature>
```

## ワークフロー

1. **コードベース分析**
   - ルート定義ファイルを Read/Grep で特定し、全エンドポイントを列挙
   - コンポーネント構造を Glob で探索し、UI フローを把握
   - 既存テストファイルを確認し、未カバー領域を特定

2. **データフロー分析**
   - フォーム送信 → API リクエスト → レスポンス → UI 更新
   - 各ステップでの検証ポイントを特定

3. **テストシナリオ設計**
   - 4層検証（API/データ/UI/Visual）を必ず含める
   - エッジケース（非同期操作、遅延レスポンス、エラーハンドリング）

## 出力形式

```markdown
# テスト計画: [機能名]

## 対象ユーザーフロー
1. [フロー説明]

## テストシナリオ

### シナリオ1: [名前]

**検証ポイント（4層）**:
- [ ] API: [検証内容]
- [ ] データ: [検証内容]
- [ ] UI: [検証内容]
- [ ] Visual: [検証内容]

**偽陽性リスク**: [低/中/高]
**リスク軽減策**: [対策]

**手順**:
1. [ステップ]

**期待結果**:
- [結果]
```

## 禁止事項（MUST NOT）

- テストコードを生成してはならない（計画のみ）
