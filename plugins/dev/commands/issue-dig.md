# /dev:issue-dig - Per-Issue 曖昧点検出

Issue 候補に対して4観点で曖昧点を検出し、構造化された結果を返す。

## 使用方法

```
/dev:issue-dig <title> <scope-summary> [<acceptance-criteria>]
```

## 入力

| パラメータ | 必須 | 説明 |
|-----------|------|------|
| title | Yes | Issue 候補のタイトル |
| scope-summary | Yes | スコープ概要（含む/含まない） |
| acceptance-criteria | No | 受け入れ基準（あれば） |

## 検出観点（MUST）

以下の4観点で曖昧点を検出する:

### 1. AC検証可能性

受け入れ基準が具体的かつテスト可能か:

| チェック | 曖昧な例 | 具体的な例 |
|---------|---------|-----------|
| 定量的か | 「適切に動作する」 | 「レスポンス200を返す」 |
| 観測可能か | 「パフォーマンスが良い」 | 「応答時間が500ms以下」 |
| 完了判定可能か | 「改善する」 | 「エラー率が1%未満になる」 |

AC が未定義の場合、「AC未定義」として WARN 判定。

### 2. スコープ境界

含む/含まないが明確か:

- 隣接機能との境界は定義されているか
- 「含まない」が明示されているか（暗黙の除外は曖昧点）
- 既存コンポーネントへの影響範囲は特定されているか

### 3. 依存関係・前提条件

他 Issue/コンポーネントへの依存が特定されているか:

- 先行 Issue が必要か（順序依存）
- 外部サービス/API への依存はあるか
- 環境前提（特定ツール、設定）が明記されているか

### 4. 実装粒度

1 PR で完結可能なサイズか:

- 変更ファイル数の推定（目安: 10ファイル以内）
- 変更行数の推定（目安: 500行以内）
- サブタスク分割が必要な兆候（複数の独立した関心事を含む）

## 判定ロジック（MUST）

各観点の検出結果を集約し、総合判定を行う:

| 総合判定 | 条件 |
|---------|------|
| **PASS** | 全観点で曖昧点なし |
| **WARN** | 軽微な曖昧点あり（AC未定義、スコープの暗黙除外など） |
| **FAIL** | 重大な曖昧点あり（検証不可能なAC、未特定の依存、粒度過大） |

## 出力形式（MUST）

```
## issue-dig 結果: <title>

### 総合判定: PASS / WARN / FAIL

### 観点別結果

| 観点 | 判定 | 検出内容 |
|------|------|---------|
| AC検証可能性 | ✓/⚠/✗ | （検出内容または「問題なし」） |
| スコープ境界 | ✓/⚠/✗ | （検出内容または「問題なし」） |
| 依存関係 | ✓/⚠/✗ | （検出内容または「問題なし」） |
| 実装粒度 | ✓/⚠/✗ | （検出内容または「問題なし」） |

### 質問候補

（曖昧点がある場合のみ。各曖昧点に対する具体的な質問）

1. 〜について、具体的にはどのような条件で完了とみなしますか？
2. 〜は本 Issue のスコープに含みますか？
```

## 禁止事項（MUST NOT）

- **Issue の内容を変更してはならない**（検出と報告のみ）
- **曖昧点がない場合に無理やり指摘を作ってはならない**
- **issue-structure や issue-create を呼び出してはならない**
